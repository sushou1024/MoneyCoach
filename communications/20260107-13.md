## 2) P0：契约/枚举一致性（会直接导致客户端/服务端/LLM 互相打架）

### 2.1 OCR status 与 DB status/错误类型：建议“单一权威枚举”

你在文档里出现了 3 套“看起来差不多但不完全一致”的状态：

* OCR LLM 输出：`success | ignored_invalid | ignored_unsupported | ignored_blurry`
* API status enum：`success | ignored_invalid | ignored_unsupported | ignored_blurry | ignored_timeout`
* DB upload_images.status：你写了 `success|ignored_invalid|ignored_blurry|ignored_timeout|ignored_unsupported|error`

**建议（P0）**：明确一份“全链路权威枚举”，并规定：

* **LLM 只能输出 LLM_allowlist**（4个）
* **Backend 扩展 backend_only**（ignored_timeout）
* **DB 不要额外引入 error**（除非你要一个“系统错误/解析异常”专用状态），否则会让前端误判

推荐你统一成：

* `upload_images.status` **只允许**：
  `success | ignored_invalid | ignored_unsupported | ignored_blurry | ignored_timeout`
* 解析失败（例如 Gemini 返回非 JSON / schema 不匹配）：不要写 `status=error`，而是
  `status=ignored_invalid` + `error_reason=PARSE_ERROR`
  或者 batch-level `upload_batches.status=failed` + `error_code=EXTRACTION_FAILURE`

这样客户端 UI 也简单：**永远只看 status + error_reason**。

### 2.2 error_reason vs batch error_code：建议固定映射表

你现在的 batch-level 规则清晰（success_images 判定），但建议补一张“映射表”写进 spec（避免未来工程师自由发挥）：

* image.error_reason：

  * `NOT_PORTFOLIO | UNSUPPORTED_VIEW | LOW_QUALITY | PARSE_ERROR | null`
* batch.error_code：

  * `INVALID_IMAGE | UNSUPPORTED_ASSET_VIEW | EXTRACTION_FAILURE | QUOTA_EXCEEDED | ENTITLEMENT_REQUIRED | NOT_READY ...`

并明确：

* `UNSUPPORTED_VIEW`（image级） **在 batch 全部都是该类型时** 才会上升为 `UNSUPPORTED_ASSET_VIEW`（batch级）
* `LOW_QUALITY`（image级）不会上升为 `INVALID_IMAGE`，除非 success_images=0 且其余都 ignored_*。

这会减少“前端拿到 E02 但其实是 E04”的客服灾难。

## 4) P0：未定价/手工估值资产的“身份（asset_key）”策略要补齐

你在 prototypes 里允许：

* 不支持/无法定价资产 → 用户手输 `manual_value_usd` → 计入净资产但不参与策略/insights

**关键问题**：如果 asset_key 无法解析（symbol null / 未识别 / 纯名字），你仍然需要一个**稳定 ID**，否则：

* 资产在 UI 列表里无法稳定更新/删除（SC10 edits 会很痛）
* 去重/合并难做
* 历史快照 diff 会乱

**建议（P0）**：为 user_provided/unpriced 资产定义一个 deterministic 的 pseudo key：

* `asset_key = manual:{user_id}:{sha256(symbol_raw_normalized|platform_guess|maybe_name)[:12]}`
  或更简单：`manual:hash:{sha256(...)}`
* 并且在 `portfolio_holdings` 里允许 `asset_key` 以 `manual:` 开头
* 明确：`manual:` 类型 **永远不进入策略/insights 的候选池**

这样你整个系统不会出现“有一笔价值但没有身份”的幽灵资产。

## 6) P0：Insights 刷新任务的负载/一致性（15 分钟轮询很容易把自己打爆）

你写了：

* `insights_refresh` every 15 min
* leader lock 防止重复
* skip 最近 15m 刷新过的用户

这很好，但在真实增长时最容易出现两个问题：

### 6.1 需要“用户级抖动（jitter）”

如果你每 15 分钟整点跑，所有付费用户会在同一时刻被刷新，Binance/CMC/Marketstack 会一起打到 rate limit。

**建议（P0）**：

* 每个 user 计算一个 deterministic offset：`offset = hash(user_id) % 900` 秒
* 每轮刷新 window 内按 offset 分桶执行

### 6.2 需要“provider 级熔断/降级开关”

你在 spec 里写了 fallback（Binance 不可用就用 CMC daily），但建议加一层“全局降级状态”：

* Redis key：`degrade:binance_klines=true` TTL 5m
* 当连续失败 > 阈值，自动开启降级，避免每个用户都重复触发失败重试

这是 MVP 稳定性关键点之一。

## 7) P0：CI/CloudFormation 的两个高风险点

### 7.1 你现在把所有 secrets 作为 CloudFormation 参数注入：可行，但要注意泄露面

你在 workflow 里用 `--parameter-overrides JwtSigningSecret="$JWT_SIGNING_SECRET"` 这种方式，**NoEcho 能保护 CloudFormation 控制台显示**，但仍有两个风险：

* GitHub Actions 日志里，如果 AWS CLI 报错把参数 echo 出来，虽然 GitHub 会 mask secrets，但可能出现 **部分泄露**（尤其是 base64 JSON）
* CloudFormation change set/事件里仍有边界情况

**建议（P0）**（最小改动版）：

* 在 deploy job 里对 `aws cloudformation deploy` 命令加 `set +x`（确保 shell 不会打印命令）
* 不要在日志里打印参数值（你现在没打印，保持）

**建议（P1，更正统）**：

* 用 Secrets Manager：把第三方 key 放进一个 JSON secret（比如 `${ServiceName}-app-secrets`），ECS TaskDefinition 用 `Secrets:` 引用 ARN，而不是把明文塞进 CFN 参数。

### 7.2 CloudFormation 网络部分：你目前片段里已经出现“只建了 PrivateRouteTableA”

你贴出来的模板截断在：

* `PrivateRouteTableA` / `PrivateRouteA` / `PrivateSubnetAssociationA`
* 但还没看到 `PrivateRouteTableB` / `PrivateSubnetAssociationB`

如果你后面确实漏了 B，那么：

* PrivateSubnetB 没有默认路由 → ECS tasks 如果被调度到 B，会**完全无法访问外网 provider**（CMC/Marketstack/OER/Gemini）
* 或者你被迫只用一个私有子网，降低可用性

**建议（P0）**：确保两条私有子网都能 NAT 出网（哪怕只用一个 NAT，也要给 B 的 route table 配一条到 NAT A 的路由）。
