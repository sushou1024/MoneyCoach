### 2.2 Health score & volatility score math conflicts with PRD rubric + your own examples

#### (A) PRD rubric says **hard constraints**, but clamp (±5) prevents enforcing them

PRD (free prompt) says:

* “If single asset dominates >50% … score MUST be below 50.”

Your MVP gap says:

* backend baseline + clamp model output ±5.

Your baseline formula might easily yield >50 even when top_asset_pct is extreme (example: 80% BTC with moderate drawdown).

**Why it matters:** you’ll end up violating the PRD’s “MUST” rules, and because you clamp, the LLM can’t fix it.

**Fix (recommended): add deterministic “hard caps” after baseline** so the clamp still works but you never violate the rubric. Example rule layer:

* If `top_asset_pct >= 0.50` ⇒ `health_score = min(health_score, 49)`
* If `top_asset_pct >= 0.70` ⇒ `health_score = min(health_score, 45)` (or whatever you want)
* If `memecoin_pct >= 0.50` ⇒ `health_score = min(health_score, 39)` (optional; see memecoin classification below)

This keeps stability and respects the PRD logic.

#### (B) Your volatility_score definition does not match typical ranges (and your examples)

You define:

* `volatility_30d = stddev(daily log returns) (not annualized)`
* `volatility_score = clamp(volatility_30d * 100 * 1.5, 0, 100)`

If volatility_30d ~ 0.04 (4% daily vol), score ~ 6. That will *never* produce sample numbers like 66 or 88.

**Why it matters:** UI and risk messaging will look broken (“volatility is extreme” but score is 6).

**Fix:** pick one of these and document it:

1. **Annualize volatility_30d** for scoring only (keep daily for strategy thresholds), e.g.

   * `vol_annual = vol_daily * sqrt(365)`
   * `volatility_score = clamp(vol_annual * 100, 0, 100)` (or another scale)

2. Keep non-annualized vol, but **change scaling** to match a 0–100 band, e.g.

   * `volatility_score = clamp(volatility_30d * 1000 * k, 0, 100)` (choose k by calibration)

Also: your S01 stop-loss volatility thresholds (0.04/0.06) assume **daily** volatility. That’s a good reason to **keep two metrics**:

* `volatility_30d_daily` (strategies)
* `volatility_30d_annualized` (UI score)

#### (C) Radar “drawdown score normalization” is under-specified

You say “drawdown normalized from 90d max drawdown” but don’t define mapping to 0–100.

**Fix:** specify a deterministic mapping like:

* `drawdown_score = clamp(100 - max_drawdown_90d * 200, 0, 100)` (example)
  and calibrate the multiplier so typical drawdowns look meaningful.

## 3) Ambiguities / under-spec that will bite later (medium priority)

These don’t strictly contradict PRD, but they’re “implementation traps” where two engineers will interpret differently.

### 3.1 Pre-check classifier: model/approach is undefined

You define a contract and thresholds, but not *how it’s implemented*.

Questions your docs don’t answer (yet):

* Is it a deterministic keyword detector (OCR on headers + regex)?
* A tiny vision classifier (CLIP / MobileNet)?
* A cheap LLM call (but that defeats cost-saving)?

**Recommendation:** pick one and specify it explicitly. For MVP, “cheap and good enough” is often:

* server-side **light OCR on top header only** + keyword rules (your unsupported keywords list is already strong)

### 3.2 Gemini “confidence” field is ambiguous

You require OCR output includes `confidence`. Gemini doesn’t natively output calibrated confidence for each extracted field.

**Recommendation:** define what “confidence” means:

* Model self-reported confidence (often not reliable), or
* Deterministic confidence computed by backend (e.g., based on regex quality, numeric parsing success, symbol validation success)
  If you keep it, make it backend-generated so it’s stable and testable.

### 3.3 USD vs stablecoin vs “USD cash inside exchange” mapping needs a rule

You have:

* stablecoin list: USDT, USDC, etc
* USD-like labels: USD, CASH, BUYING POWER…
* platform_category mapping

But if a crypto exchange screenshot shows **USD** (not USDT), what is it?

* fiat cash (forex) or crypto?

**Recommendation:** add a deterministic rule:

* if platform_category is crypto_exchange/wallet and symbol_raw_normalized == “USD”, treat as **forex USD cash** unless the platform explicitly labels it as a token.

### 3.4 Manual valuations: do they affect health_score and concentration?

You say:

* user_provided values are in net worth
* excluded from strategy parameters and Insights triggers

But you don’t specify:

* whether they count in concentration (`top_asset_pct`) and cash_pct used in scoring.

**Recommendation:** include user_provided values in:

* net worth
* concentration & cash_pct
  …but exclude from:
* volatility/corr/drawdown metrics (since no OHLCV)

Otherwise you can get weird outcomes like “huge manual asset doesn’t affect concentration risk”.

### 3.5 Delta updates without recomputing the plan: where is plan state stored?

You say delta updates:

* don’t recompute optimization_plan
* “advance next actions (next DCA time, next safety order…)”

But there’s no explicit plan state model/table.

**You will need** a runtime state per plan/strategy, e.g.:

* S02: next_order_index
* S03: activated_at, peak_price_since_activation
* S05: next_execution_at

**Recommendation:** add a `plan_state` JSONB per `(user_id, plan_id)` or store derived state from transactions + snapshot, but specify which approach.

### 3.6 Dedupe override UX: allowing “include duplicates” can double-count

SC10 allows restoring duplicates. If user toggles include, aggregation will double count.

**Recommendation:** add explicit UX + backend warning:

* “Including a duplicate screenshot may double-count holdings. Only include if this screenshot is a different account.”

### 4.2 A few practical infra under-spec / footguns

Not “wrong”, but worth calling out:

* **Redis** is a single-node `CacheCluster` (no replication/failover). Fine for MVP, but document it as “non-HA”.
* **Secrets handling**: most secrets are passed as CFN parameters → ECS env vars (readable via ECS APIs to privileged users). Safer pattern is ECS “Secrets” from Secrets Manager/SSM. Not required, but note it.

* **HTTPS** is conditional on ACM cert. For production, you probably want to require it (or at least warn in docs).

## 5) Environment variables: sufficiency + “required vs optional” cleanup

### 5.1 Are env vars sufficient?

For the features described, yes—**you cover the major providers**:

* Gemini (LLM), Resend (email), CMC, Marketstack, OpenExchangeRates, Binance, Stripe, Apple, Google.

One caveat:

* Your iOS receipt verification approach is implicitly “shared secret + receipt verification”. If you later move to **App Store Server API (JWT)**, you’ll need additional Apple key material. Right now your env set suggests you’re *not* using that path (which is fine).

### 5.2 Which env vars are truly “core required” vs “conditional”?

Right now, your template **always injects** all payment env vars into the container. That’s okay operationally, but it tends to confuse new engineers because it looks like “everything is required”.

Here’s a cleaner classification you can put in docs (and optionally enforce in code).

#### Always required (core backend runs)

* `JWT_SIGNING_SECRET`
* `GEMINI_API_KEY`
* `CMC_PRO_API_KEY`
* `MARKETSTACK_ACCESS_KEY`
* `OPEN_EXCHANGE_APP_ID`
* `RESEND_API_KEY`
* `RESEND_FROM_EMAIL`
* `DATABASE_URL` *(injected by CFN)*
* `REDIS_URL` *(injected by CFN)*
* `OBJECT_STORAGE_BUCKET` *(injected by CFN)*
* `OBJECT_STORAGE_REGION` *(injected by CFN)*

#### Recommended defaultable (should not be required)

* `BINANCE_API_BASE_URL` (default to `https://api.binance.com`)

#### Required only if Google login enabled

* `GOOGLE_ALLOWED_CLIENT_IDS`

#### Required only if Stripe payments enabled (Android/PWA in your doc)

* `STRIPE_SECRET_KEY`

### 5.3 Concrete ways to reduce confusion (recommended)

Pick one:

**Option A (best dev UX): introduce feature flags**

* `ENABLE_STRIPE=false|true`
* `ENABLE_APPLE_IAP=false|true`
* `ENABLE_GOOGLE_PLAY=false|true`
* `ENABLE_APPLE_WEB_OAUTH=false|true`
  Then validate env vars conditionally at boot:
* If enabled and missing → fail fast with a clear error.

**Option B (doc-only): keep envs but label them “conditional”**

* Your backend spec already hints “required when channel enabled”.
* Make it explicit in a single “Env Vars” table in one canonical doc.

**Option C (CloudFormation cleanup): give defaults for all conditional params**
Right now, many payment params have no defaults. You *are* passing them from CI anyway, but defaults make it clearer they’re optional.

Examples worth giving `Default: ""`:

* `StripePriceIdWeekly`, `StripePriceIdYearly`
* Apple IAP IDs
* Google Play IDs

Then in code, treat empty string as “disabled”.
