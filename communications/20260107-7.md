## 2) P0 issues / inconsistencies worth fixing now

### P0-A) **Header masking vs server-side pre-check (major)**

You wrote:

* Client: “best-effort header masking before upload (fixed top region)”
* Pre-check classifier: “run lightweight OCR on the top header region + keyword rules” to detect Futures/Options/Margin etc.

If the client masks the same region your server needs to read, you lose the strongest signal for:

* futures/options tabs (`合约/永续/杠杆/仓位` etc)
* platform UI context
* sometimes even the *market/asset list header*

**Why this matters:** you’ll misclassify unsupported screens as supported, then waste Gemini calls, and you’ll also miss good early rejection of junk images.

**Clean fix (recommended):**

* Upload **raw** screenshot to S3 (treat as sensitive).
* Server pipeline:

  1. **Pre-check** on raw image (internal only; never log text).
  2. Apply **server-side masking** (fixed region) to produce a “LLM-safe” variant.
  3. Send masked variant to Gemini.

This keeps your MVP privacy story intact (“we mask before sending to model”), while preserving pre-check reliability.

**If you must keep client-side masking:**

* Change masking geometry so it **does not cover navigation/tab labels**.
  Example: mask only account ID/phone/email sub-areas, not full top header strip.

---

### P0-B) Insights `type` enum casing is inconsistent

In prototypes you show:

```json
"type": "Market Alpha"
```

But backend spec + DB schema uses:

* `portfolio_watch | market_alpha | strategy_execution`

Pick one canonical format for API contracts (I strongly recommend the lowercase snake-case enum), and keep “Market Alpha” as a **UI label only**.

**Fix pattern:**

* API: `type: "market_alpha"`
* UI: map to display string `"Market Alpha"`

Otherwise you’ll end up with:

* hard-to-debug filter mismatches
* broken analytics dimensions
* inconsistent TTL/dedupe keys

---

### P0-C) Risk id naming: `id` vs `risk_id`

Preview uses `identified_risks[].id`
Paid uses `risk_insights[].risk_id`

That can work, but it’s friction you don’t need.

**Simplest improvement:**
Standardize on `risk_id` everywhere:

* preview: `identified_risks[].risk_id`
* paid: `risk_insights[].risk_id`

You can still display “risk_01” etc.

This matters because you explicitly want: “1:1 expansion of preview risks” and you’ll probably join against `report_risks` anyway.

### P0-D) “80% of users achieve long-term profitability” claim

You already noted it as placeholder in SC01. This is a compliance/legal/ASO risk if it ships.

**Recommendation:** replace with something you can support immediately (or remove):

* “Used by investors managing $X assets” (only if true)
* “Institutional-style risk scoring”
* or just keep it as product narrative without numeric claims

---

### P0-E) CloudFormation/CI parameter optionality for payments

Your stack currently *requires* a lot of payment-related parameters (Stripe IDs, Apple product IDs, Google IDs). If any are missing/empty, deployments can become fragile.

**Two good patterns:**

1. **Feature flag parameters**: `EnableStripe`, `EnableAppleIAP`, `EnableGooglePlay`
   Use CloudFormation `Conditions` so you only inject env vars when enabled.
2. Put payment provider secrets in **Secrets Manager** and reference them only if enabled.

For an MVP, you can still keep it simple, but make sure a “crypto-only internal test deployment” doesn’t require all the store config to exist.

## 3) Backend spec quality: what’s strong, and what to tighten

### Strong decisions you made (keep them)

* **LLM never does math**: USD conversion, net worth, strategy parameters are backend code. ✅
* **Preview → Paid invariants**: health_score and risks are locked, and paid report justifies them. ✅
* **Market snapshot**: preview + paid share `market_data_snapshot_id` (prevents valuation drift). ✅
* **User-provided valuation isolation**: included in net worth but excluded from triggers/plan params. ✅
* **Delta updates do not mutate the original report**: avoids “moving target” paid content. ✅

### Tighten these areas

#### 3-A) Idempotency: expand it to *all* mutating POSTs

You say idempotency is required for money/report-related endpoints. In practice, you want it for:

* `POST /upload-batches`
* `POST /upload-batches/{id}/complete`
* `POST /upload-batches/{id}/review`
* `POST /reports/{calculation_id}/paid`
* `POST /billing/receipt/*`
* `POST /insights/{id}/execute`
* `POST /waitlist`

Because mobile clients retry aggressively on flaky networks.

**One extra guard:** for delta updates, also take a short Redis lock:

* `lock:portfolio:{user_id}` (you already listed this)
  so two executes don’t race and create inconsistent snapshots.

#### 3-B) Dedupe robustness

Fingerprint-on-OCR is good, but OCR output can jitter (rounding, missing rows, partial extraction).

Low-effort improvement:

* Add a **perceptual hash** (pHash/aHash) or even “image bytes hash after normalize/compress” as a *secondary* duplicate hint.
* Use it only to *warn*, not auto-exclude.

#### 3-C) Volatility annualization: stocks vs crypto

You annualize with `sqrt(365)` globally. That will overstate stock vol vs common convention (sqrt(252)).

For MVP it’s fine if you treat this as an internal heuristic, but it will affect:

* volatility_score
* health_score penalties
* cross-asset comparability

**Easy fix:** use:

* crypto: `sqrt(365)`
* stocks: `sqrt(252)`
* forex cash: not applicable

#### 3-D) Timezone & quota day boundary

Your precedence chain is good:
`user_profiles.timezone -> upload_batch.device_timezone -> UTC`

Two practical edge cases:

* If user changes timezone later, your stored “quota day boundary” may shift.
* S05 `next_execution_at` stored in UTC: if timezone changes, should you shift the schedule?

Minimal MVP approach:

* Keep quota boundary based on **user_profiles.timezone if set**, else fallback.
* For S05, only reschedule future executions when user explicitly changes timezone in Settings.

## 4) CI + CloudFormation audit

### What looks correct

* OIDC to AWS via `aws-actions/configure-aws-credentials@v4` ✅
* ECR repo auto-create ✅
* CloudFormation parameters cover the backend’s env vars (JWT, Gemini, providers, payments) ✅
* VPC with private subnets for DB/Redis ✅
* S3 bucket encryption + lifecycle ✅
* DB credentials in Secrets Manager ✅

### High-impact improvements (infra)

#### 4-A) Secrets in ECS task definition

Even with `NoEcho`, env vars in ECS task definitions are visible to anyone who can `DescribeTaskDefinition`.

Best practice:

* Use `ContainerDefinitions[].Secrets` with Secrets Manager/SSM parameters.
* Keep only non-sensitive configs in `Environment`.

This is not required to ship MVP, but it’s a big security hygiene win.

#### 4-B) NAT Gateway cost / complexity

You built a “proper” private-subnet + NAT design. It’s correct, but NAT is a real fixed cost.

If you want a cheaper MVP:

* Put ECS tasks in **public subnets** with `AssignPublicIp: ENABLED`
* Keep DB/Redis private
* Allow inbound to tasks only from ALB SG

That removes NAT entirely.

#### 4-C) Logging configuration

Your template creates a `LogGroup`, but your container definition snippet doesn’t show `LogConfiguration`.

Make sure your ECS container has awslogs configured; otherwise debugging OCR failures and provider errors will be painful.
