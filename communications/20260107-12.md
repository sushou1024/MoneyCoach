## 2) The biggest spec-level mismatch to resolve (important)

### A) “Tab 1 is default home” vs “Insights is paid-only”

In PRD Module 5, Tab 1 (Insights) is described as *App 首页默认 Tab*.
In your storyboards, after auth you go **SC08 Scan Home** (Tab 2), which is a sensible MVP adjustment because free users would otherwise land on a locked paywall screen immediately.

**What you should do explicitly (pick one and make it consistent in all docs):**

1. **Dynamic default tab** (recommended):

   * If **paid entitlement active** → default tab = **Insights**
   * If **free** → default tab = **Scan**
2. Or keep PRD behavior: always default to Insights, but that’s conversion-heavy and risks higher bounce.

Right now your prototypes imply (1), but the PRD text implies (2).
This isn’t just “docs nit”: it affects onboarding conversion and app store review optics.

## 3) Spec drift / ambiguity that will bite later

### A) OCR “confidence” field: decide where it lives

You say:

* “confidence is backend-computed; do not rely on model”
  but your example payloads include `confidence` inside `assets[]`.

That’s fine, but you need **one of these**:

* Option 1: **Remove confidence from the Gemini output schema entirely** (cleanest), and attach it in backend responses.
* Option 2: Keep it in the API response but explicitly state: `confidence_source = backend`.

If you leave it ambiguous, someone will accidentally treat it as model confidence later.

---

### B) Dedupe fingerprint timing vs canonical asset_key availability

You want to compute per-image fingerprint immediately after OCR, before SC10.

But your strongest fingerprint uses `asset_key` (cmc_id / exchange_mic), which typically **isn’t resolved until after symbol validation**.

You already have a fallback (“use symbol_raw if asset_key missing”), which is good. What I’d add explicitly:

* **Fingerprint v0** (pre-review): platform_guess + row_count + (symbol_raw_normalized:amount)
* **Fingerprint v1** (post-resolution): platform_guess + row_count + (asset_key:amount)

…and only use v0 for **warning + “likely duplicate”** before SC10, then confirm v1 after resolution.

This avoids false positives like `ABC` stock vs `ABC` token collisions.

### C) Quota day boundary: store “usage date” in user timezone explicitly

You currently have:

* Quota reset uses user local day; if timezone unknown, use UTC.
* DB table: `quota_usage.usage_date (date)`

If you store a `date` without storing the timezone used to compute it, you can get incorrect resets when:

* user changes timezone,
* DST changes,
* device timezone differs from profile timezone.

**Practical MVP-safe approach:**

* compute `usage_day = current_time at timezone_precedence` → YYYY-MM-DD
* store:

  * `usage_day` (date)
  * `timezone_used` (text)
  * optionally `window_started_at_utc`

This prevents “free quota got eaten twice today” support tickets.

### D) Paid conversion flow: ensure preview snapshot is immutable

You correctly describe:

* valuation snapshot is immutable for preview + paid
* strategy params computed using that snapshot to avoid drift

Just make sure the backend enforces:

* **paid report must reference the same `market_data_snapshot_id` as preview**
* **plan parameters are computed off that snapshot only**, never “latest quote”

You wrote this, but it’s worth enforcing at the DB schema + code level (foreign keys + assertions), because valuation drift is the #1 “your app is wrong” complaint.

---

### E) “+15% APY” projection language is risky

Your PRD and prototype include explicit performance claims (“预计年化提升 +15%”).

Even with disclaimers, this can cause:

* higher app store review scrutiny
* consumer complaint / regulatory sensitivity depending on jurisdictions

If you want “conversion juice” without explicit yield claims, a safer MVP pattern is:

* “Potential risk reduction”
* “Projected drawdown reduction”
* “Improve discipline / reduce concentration”
* or “scenario-based projection” with wording like “simulated” and no precise %.

You can keep the number internally for A/B tests, but I’d be careful about shipping it verbatim.
