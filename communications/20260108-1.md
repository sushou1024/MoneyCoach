## 2) P0 issues (fix these now to avoid product/engineering drift)

### P0-A) Strategy allowlist leak: your prototypes still reference **S12** in Market Alpha copy

In your **PRD** and later your prototypes, Market Alpha examples include:

* “暴跌/超卖 (**S12 均值回归**)… RSI < 30…”

But your MVP Gap + your prototypes say **S01–S05 only**.

**Why this matters:**
Even if you keep `strategy_id=null` for those signals, the UI copy and spec still “teaches” engineers and designers that S12 exists in MVP. That tends to leak into: enums, analytics, future toggles, QA expectations, and even paywall claims.

**Fix (recommended):**

* In MVP, make Market Alpha signals **strategy-less** (no strategy ID reference) and never mention S12.
* Rename the copy conceptually to “Quant Signal” / “Market Anomaly” instead of “S12 strategy”.

**Concrete doc patch:**

* Replace any “S12” in PRD/prototypes with:

  * “📉 Opportunity: BTC RSI < 30 on 4h, touching lower Bollinger band. Watch for rebound confirmation.”

And keep `strategy_id=null` for Market Alpha items in the feed schema.

### P0-B) Timezone precedence contradiction: quota reset rule is inconsistent in your docs

You have two conflicting statements:

1. In prototypes:

> “Daily quota reset uses user local day; **if timezone unknown, use UTC**.”

2. Elsewhere (and in backend spec):

> “Timezone precedence: user_profiles.timezone -> upload_batch.device_timezone -> UTC. Applies to quota boundary…”

These aren’t the same.

**Why this matters:**
Quota is a paywall lever. If you let `device_timezone` drive quota when `user_profiles.timezone` is empty, users can **flip device timezone to get extra “days”**.

**Fix (recommendation):**

* **Quota boundary**: `user_profiles.timezone ?? 'UTC'`
  (ignore device_timezone for quota)
* **Scheduling (S05 next_execution_at)**: `user_profiles.timezone ?? upload_batch.device_timezone ?? 'UTC'`
  (device timezone is fine for scheduling convenience)

If you *must* use device timezone for quota, then lock it:

* store `timezone_used` on the first quota record of the day and **do not allow changes until next reset**.

### P0-C) S02 Martingale base order sizing can produce < $20 orders even when idle_cash_usd >= 20

Your current S02 spec:

> `base_order_usd = min(clamp(idle_cash_usd * 0.05, 20, 2000), idle_cash_usd / series_sum)`

Edge case:

* `idle_cash_usd` slightly > 20
* `idle_cash_usd / series_sum` can be **< 20**, making `base_order_usd` < 20

That’s inconsistent with “min order size” assumptions and will look broken in UI (and possibly in exchanges later).

**Fix (deterministic and simple):**
Use a clamp **after** min selection, or clamp the final result:

* Option 1 (simple):

  * `raw = min(idle_cash_usd * 0.05, idle_cash_usd / series_sum)`
  * `base_order_usd = clamp(raw, 20, 2000)`
  * If `base_order_usd * series_sum > idle_cash_usd`, then reduce `base_order_usd = idle_cash_usd / series_sum` and if that’s < 20, skip S02.

* Option 2 (more explicit):

  * Compute `max_affordable = idle_cash_usd / series_sum`
  * If `max_affordable < 20` → skip S02
  * Else `base_order_usd = clamp(idle_cash_usd * 0.05, 20, min(2000, max_affordable))`

This keeps behavior stable and exchange-compatible.

### P0-D) HTTPS should not be optional for production mobile auth

Your CloudFormation parameter `AcmCertificateArn` is optional (“leave empty to disable HTTPS listener”).

For a real mobile app exchanging JWTs/refresh tokens, running HTTP is a non-starter beyond local/dev.

**Fix (recommended):**

* Keep the condition for dev stacks, but:

  * In “prod” stacks: require a cert (or fail deployment if empty).
  * Or always create HTTPS listener and make HTTP redirect to HTTPS.

At minimum, make sure the production deploy path *never* ships without TLS.

## 3) P1 improvements (very worthwhile; reduce future “MVP debt”)

### P1-A) Clarify “Market Alpha” vs “Strategy Execution” semantics in API + analytics

Right now you have:

* Feed item schema: `type=market_alpha|portfolio_watch|strategy_execution`
* Optional `strategy_id` must be S01–S05 if present

This is good, but add one explicit invariant to prevent future drift:

* **market_alpha items MUST NOT have strategy_id** in MVP (always null)
* **strategy_execution items MUST have plan_id** (since execution is tied to locked plans)

This avoids the “S12 leak” problem structurally.

---

### P1-B) Don’t store huge provider payloads in one `market_data_snapshots.provider_payloads` blob without limits

Storing full raw payloads is great for auditability, but the blob can become large fast (especially OHLCV arrays).

**Recommendation:**

* Keep `market_data_snapshots` minimal:

  * snapshot metadata + provider request fingerprints + quote timestamps
* Store large payloads either:

  * in `market_data_snapshot_items.raw_payload` (already in your schema), **or**
  * in S3 referenced by snapshot id (cheaper + easier to prune)

If you keep payloads in Postgres:

* enforce size caps and prune OHLCV bodies.

---

### P1-C) Tighten numeric types (Postgres `numeric`) to avoid silent precision drift

Right now everything is `numeric` (fine), but specifying precision saves you from future bugs:

Typical safe choices:

* `amount`: `numeric(36, 18)` (crypto can be tiny fractions)
* `price_usd`: `numeric(36, 18)` for crypto, but you can still store at 18 and round for UI
* `value_usd_priced`: `numeric(36, 6)` or `numeric(36, 8)` (6–8 decimals is plenty)
* `pnl_percent`: `numeric(10, 6)` (store -1.000000 to +10.000000 etc)

This also makes “rounding rules” enforceable.

---

### P1-D) Make “manual/user_provided” holdings affect **health score** more explicitly

You currently:

* include `user_provided` in net worth and concentration/cash metrics
* exclude from vol/corr/drawdown metrics

This is okay, but you should document one more rule to keep results intuitive:

* If `priced_coverage_pct < 0.60`:

  * force one of the three risks to be “Pricing Coverage / Data Limitation” **or**
  * force severity of risk_03 to be at least Medium and mention limitations in teaser and paid verdict.

You already have the “metrics_incomplete” hook; I’d just make the outcome deterministic so the product feels consistent.

---

### P1-E) CI safety: prevent parallel deploys + empty changeset failures

In `.github/workflows/ci.yml`, add:

* **Concurrency** so two pushes don’t race deploy:

  * `concurrency: group: app-backend-${{ github.ref }}`
* Add `--no-fail-on-empty-changeset` so deploy doesn’t fail when nothing changed.

Also consider switching to `docker/build-push-action` with build cache to speed iteration.

---

### P1-F) CloudFormation secret hygiene

Passing secrets as CFN parameters works (with `NoEcho: true`), but it’s operationally painful because:

* rotating a key means redeploying with parameter overrides
* it’s easy to accidentally diff/print something later

**Recommendation:**

* Store third-party API keys in **Secrets Manager** (like you already do for DB credentials).
* Reference them from ECS TaskDefinition via `Secrets` rather than environment variables.

Even in MVP, this is usually a net win.
