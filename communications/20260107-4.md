## 3) Ambiguities / under-specifications that can break engineering or UX

These aren’t strict contradictions, but they’re “you will fight this later” items.

### (1) Active Portfolio, portfolio snapshots, delta updates, and reports are currently semantically muddled

You state:

* Active Portfolio becomes active after SC10 confirm ✅
* Delta update: “apply delta update to Active Portfolio and create a new snapshot” ✅
* “Delta updates do not recompute optimization plan; paid report stays locked” ✅

But you don’t fully specify what this means for:

* `active_portfolio_snapshot_id`
* “My Reports” list showing [Active]/[Archived]
* whether a delta-update snapshot appears as a “new report item” or not
* whether the paid report is now tied to an archived snapshot (very likely with your DB model)

**Why this matters**: your schema ties `reports.portfolio_snapshot_id` to a snapshot. If delta updates create a new snapshot and you mark it active, your paid report is now pointing at the *old* snapshot (which is presumably archived). That makes “Active” labeling ambiguous.

**Concrete spec you should add** (one of these two models):

**Model A — Reports are tied to Scan Snapshots; Active Portfolio is a mutable state**

* Keep `portfolio_snapshots` for scan events only (SC10 confirm).
* Maintain a separate `active_portfolio_state` or `active_portfolio_holdings` that can be updated by delta updates without creating a new “snapshot.”
* Delta updates write transaction logs and mutate the active state.
* Reports remain tied to the scan snapshot and UI shows “Report based on snapshot date X; holdings updated Y.”

**Model B — Everything is snapshot-based; delta updates create snapshots, but reports remain tied to original snapshot**

* Delta updates create a new snapshot and set it active.
* Reports remain tied to the original snapshot.
* UI must clearly show:

  * “Active Portfolio (latest calibrated)” and
  * “Report Snapshot (the one your plan was generated from)”
* Also define whether delta update snapshots appear in “My Reports.” (I’d recommend **no**; they’re not full reports.)

Right now your docs imply Model B but your UX copy and “My Reports” UX imply Model A. You need to pick and specify.

### (2) Metric definitions are incomplete (volatility_30d, drawdown_90d, correlation)

You defined formulas for health score penalties that consume:

* `volatility_30d`
* `max_drawdown_90d`
* `avg_pairwise_corr`
* plus alpha score derived from 30d returns vs benchmark

But you don’t specify:

* return frequency (daily? 4h? mixed?)
* whether volatility is annualized or raw std-dev
* correlation window, frequency alignment (crypto daily vs stocks EOD)
* missing-data handling (what if an asset lacks history?)
* minimum data points required to compute (else fallback)

This will lead to inconsistent “health_score drift” across assets and markets.

**Recommendation**: add a single “Market Metrics Definition” section with:

* `returns`: log returns vs simple returns
* `volatility_30d`: stddev(daily_returns over last 30 trading days), annualized or not (choose one)
* `max_drawdown_90d`: computed on daily close series over last 90 calendar days (or trading days)
* `avg_pairwise_corr`: correlation of daily returns over 90d between top N holdings (define N and weighting)
* missing OHLCV: fallback rules (skip holding from corr calc but keep valuation)

## 4) CI + CloudFormation: do they match the docs and architecture?

### ✅ Generally consistent

Your CI pipeline, CloudFormation, and backend spec align on:

* ECS Fargate behind an ALB
* RDS Postgres + ElastiCache Redis
* S3 uploads bucket with 30-day lifecycle
* Env var naming matches the backend spec’s list
* Health check path `/healthz` matches the template

So the “shape” is consistent.

### ⚠️ Important missing infrastructure details vs your backend spec

#### (A) HTTPS/TLS is missing (this will break Stripe + most OAuth and webhooks in real environments)

Your ALB only listens on **HTTP :80**.
For production:

* Stripe webhooks generally expect HTTPS endpoints (and OAuth redirect URIs also usually require HTTPS).
* Apple/Google server notifications also expect HTTPS.

**Fix** (recommended): add HTTPS:

* ACM certificate
* ALB listener on 443
* Redirect 80 → 443

This is not just “nice”; it’s essential for payments/webhooks.

#### (B) PWA direct-to-S3 uploads will fail without S3 CORS rules

Your bucket has no CORS configuration.
If your PWA uploads directly to S3 using presigned PUT URLs, the browser will block it without CORS.

**Fix**: add `CorsConfiguration` on `UploadsBucket` with allowed methods `PUT, GET, HEAD`, allowed headers `*` (or explicit), allowed origins set to your PWA domains.

Mobile native apps don’t need CORS, but your docs explicitly include PWA.

#### (C) Background jobs / scheduling are not reflected in infra

Your backend spec defines jobs like:

* `insights_refresh` scheduled every 15 minutes
* OCR jobs, pricing jobs, report generation jobs

CloudFormation deploys a single ECS service, but doesn’t define:

* a worker service
* scheduled ECS task
* EventBridge rule
* any queue besides Redis

This can still be valid **if** you run a Redis-based queue and run workers inside the same ECS task process, but you should explicitly say so.

**Fix**: document “job execution model”:

* Option 1: “Monolith does both HTTP + workers in same container (BullMQ worker in same runtime).”
* Option 2: “Separate worker ECS service (same image, different command).”
* Option 3: “EventBridge scheduled tasks for insights_refresh.”

Right now it’s under-specified and creates operational uncertainty.
