## 2) Biggest PRD ↔ Prototypes inconsistencies (needs explicit resolution)

### A) iOS payments: “Stripe + IAP simultaneously” vs “Apple policy may require hiding Stripe”

* **PRD MVP Gap #5**: says iOS/Android both support Stripe + native payments.
* **Prototypes**: includes a note: “Platform policy may restrict external checkout on iOS; if required, hide Stripe on iOS.”

**Why this matters:** this isn’t just an implementation detail; it affects your entire conversion funnel + QA + app store risk.

**Recommendation:** pick one of these and reflect it consistently in *both* documents:

1. **“App Store-first MVP”**: iOS only IAP; Android supports Stripe + Play; PWA supports Stripe.
2. **“All-platform Stripe MVP”**: go PWA-first; native apps later.
3. **“Dual payments”**: only if you have a compliant approach (but treat this as a product/policy decision, not a footnote).

At minimum, update the prototypes doc to mark this as a **product decision** rather than an FYI note.

---

### B) “Active Portfolio becomes active when?”

* **PRD flow** (step 7 in your narrative): implies *after clicking Auto‑Execute / Waitlist* the scanned portfolio is marked **Active** and Insights start.
* **Prototypes**: “portfolio becomes Active after SC10 confirm of a valid scan (free or paid).”

These are different. Your prototypes choice is more practical (Insights can start earlier), but it’s a deviation.

**Recommendation:** decide and document the canonical rule:

* Option 1 (better UX / retention): **Active at SC10 Confirm** (recommended).
* Option 2 (more “paid-leaning”): Active only after paywall success.
* Option 3 (PRD literal): Active only after Auto‑Execute click (weak; delays retention loop).

If you keep the prototypes behavior, update PRD language in step 7 to avoid future “why is Insights running for free users?” confusion.

## 4) Ambiguities & under-specifications that will cause build churn

### 4.1 Data sources & valuation snapshot consistency (high risk)

You do a good job saying “snapshot must match between preview and paid.” What’s missing is the operational definition:

* **What is `market_data_snapshot_id`?**
  Is it:

  * a DB row containing serialized quotes for all assets?
  * just a timestamp?
  * a hash over provider payloads?

* **Multi-provider drift:**
  You value crypto via CMC but compute intraday indicators via Binance. That’s fine, but you need to specify which price feed drives:

  * valuation numbers on report (CMC)
  * strategy parameter computation (must match preview snapshot)
  * live trigger checks in Insights (likely live market feed)

**Suggested spec addition:** define 3 distinct “price contexts”:

1. **Valuation snapshot price** (immutable; used for report numbers + strategy params)
2. **Live trigger price** (used for alerts; acceptable small drift)
3. **Chart/indicator price series** (Binance/Marketstack series)

Then document expected mismatch tolerances (e.g., “trigger if price <= stop_loss * 1.005” already starts to do this; extend it consistently).

---

### 4.2 Pre-check classifier location and contract

PRD implies local/lightweight to avoid token waste; prototypes says **server-side** before Gemini.

That’s acceptable, but you need to pin down:

* input: full image? compressed? blurred?
* output: {is_portfolio_screenshot, confidence, reason_code}
* thresholds: what’s “pass” vs “fail”
* what “unsupported view” means (futures/options/margin screens)

Otherwise, you’ll get random false rejects and poor supportability.

---

### 4.3 Symbol ambiguity & “USD-like” mapping needs a hard rule set

You have good intent (“map USD to cash if UI indicates base currency; otherwise stablecoin”). But it’s under-specified:

* What counts as “USD-like”? (`USD`, `US Dollar`, `$`, `Cash`, `Buying Power`?)
* Stablecoin list: `USDT/USDC/DAI/TUSD/...`?
* How do you determine “source UI indicates base currency”? (platform_guess? keywords like “Buying Power”?)
* How do you handle broker cash vs crypto exchange cash vs wallet?

**Recommendation:** add a deterministic mapping table + platform heuristics.

---

### 4.4 Strategy parameter computation is referenced but not fully pinned for MVP

You correctly say backend computes parameters deterministically, but the prototypes doc still leaves the actual formulas ambiguous for several MVP strategies:

* **S01**: your strategy tech spec uses *support_levels* and has “adjust to support below 2%.”
  Prototypes doc does not specify how support levels are computed (from OHLCV? timeframe? method?).

* **S03** activation_price + callback_rate: not specified in prototypes doc.

* **S02** safety_orders[] pricing ladder and scale: not specified in prototypes doc.

If your intention is “these formulas live in the strategy technical spec doc,” then the prototypes doc should explicitly reference the exact sections/files and define a **minimum contract**:

* input fields required (avg_price? pnl? cash? volatility?)
* what happens when missing (you have some rules—good)
* deterministic rounding rules (important for UX consistency)

Otherwise devs will implement “best guesses.”

### 4.5 Health score + volatility score are LLM-generated but not constrained enough

Right now:

* You use strict JSON schema and “consistency is law” between preview and paid. Good.
* But you also let LLM generate `health_score` and `volatility_score` with no deterministic anchors.

**Risk:** scores will be unstable across similar portfolios; users will notice.

**Recommendation:** even if the LLM “decides,” add:

* a **feature vector** computed in backend (concentration %, cash %, #assets, top asset share, realized/unrealized drawdown proxies, etc.)
* instruct LLM that score must be consistent with those features (or even compute a baseline score in code and let LLM adjust within ±X)

This is not required by PRD, but it’s the most common “AI scoring” failure mode in production.
