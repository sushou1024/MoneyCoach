## 4) Major under-specifications (areas that need tightening)

These are the spots where engineering will either guess—or build something inconsistent with your intended “Prism” behavior.

### 4.1 Upload/OCR result contract is too loose

You say:

* “Output: per-image extraction results, unrecognized_items…”
  But don’t define the **schema** and status semantics.

**You need a per-image status model** to resolve PRD ambiguity. Example:

```json
{
  "upload_batch_id": "ub_123",
  "images": [
    {
      "image_id": "img_1",
      "status": "success | ignored_invalid | ignored_blurry | error",
      "platform_guess": "Binance | Metamask | Robinhood | Unknown",
      "assets": [
        {
          "symbol_raw": "Tether",
          "symbol": "USDT",
          "asset_type": "crypto | stock | forex",
          "amount": 5000,
          "value_display": 5000,
          "display_currency": "USD",
          "confidence": 0.92,
          "avg_price": null,
          "pnl_percent": null
        }
      ],
      "warnings": ["PII_REDACTED", "CURRENCY_GUESS"]
    }
  ],
  "summary": {
    "success_images": 2,
    "ignored_images": 1,
    "failed_images": 0
  },
  "error": null
}
```

Then define batch-level behavior:

* If `success_images >= 1`: overall status success.
* If `success_images == 0` and all are `ignored_invalid`: return `INVALID_IMAGE`.
* If `success_images == 0` and all are `ignored_blurry/error`: return `EXTRACTION_FAILURE`.

Right now this is ambiguous in your prototypes doc.

### 4.2 Symbol validation across crypto/stocks/forex needs tie-break rules

You state:

* Validate crypto via CMC map
* Stocks via Marketstack tickers
* Forex via OER currencies

But many real-world edge cases exist:

* Same string could exist in multiple domains (“ONE”, “NEAR”, “MATIC” unlikely as stocks, but collisions happen in general)
* “USD” could mean fiat cash, stablecoin, or base currency display
* Some broker screenshots may show company names not tickers

**Tie-break rule missing**:

* Prefer platform_guess context (Robinhood ⇒ stock, MetaMask ⇒ crypto)
* If platform is unknown: use heuristic + prompt user confirm

Add a small deterministic decision policy to avoid weird LLM-driven classification.

### 4.5 Strategy parameterization responsibilities are unclear (LLM vs code)

You do a great job insisting:

* “Net worth and USD conversions computed in code”
* “RSI/Bollinger computed in code”

But for strategies, your doc doesn’t clearly state:

* Which strategy parameters are computed deterministically (code)
* Which are allowed to be generated by the LLM
* How to enforce schema validity

Given your strategy library includes explicit Python-like parameter calculations (S01/S04/S05 etc), MVP should likely:

* Compute core parameters in code (deterministic)
* Let LLM generate narrative + rationale only

Otherwise you’ll get unstable outputs and mismatched backtests.

**Concrete fix**: For each allowed strategy ID, define a JSON schema like:

* S01 params: `stop_loss_pct`, `stop_loss_price`, `adjustment_rules[]`
* S04 params: `layers[]`, `adjustment_rules[]`
* S05 params: `amount`, `frequency`, `calendar[]`

And say: “Backend computes params, LLM only fills copy fields.”
