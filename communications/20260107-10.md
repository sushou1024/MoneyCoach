## 2) P0 inconsistencies / risks to fix before building too far

### P0-A) OCR per-image status enum is inconsistent (and will break clients)

* Prototypes: OCR contract status = `success | ignored_invalid | ignored_unsupported | ignored_blurry`
* But later: ‚Äúbackend marks `ignored_timeout`‚Äù and also mentions `error`

**Why this matters:**
Your mobile UI (SC10 review) and your polling responses will eventually see `ignored_timeout`. If the client SDK enums don‚Äôt include it, you‚Äôll either crash, or silently drop images.

**Fix (recommended):**

* Make `ignored_timeout` a **first-class status** in API responses for upload batches (holdings OCR).
* Keep Gemini response enum the same (4 statuses), but allow backend to *override* to `ignored_timeout`.

**Concrete changes:**

* In `prototypes/quanta-v1-prototypes.md` and `backend spec`, update:

  * OCR status enum ‚Üí add `ignored_timeout`
  * Batch resolution rules: treat `ignored_timeout` as ‚Äúignored‚Äù (same bucket as blurry/invalid for invalid-image detection)

## 4) CloudFormation / CI review (P0-P1 infra correctness)

### P0-E) Your CloudFormation file in the message is truncated at `TaskRole`

So I can‚Äôt verify the ECS cluster/task/service/ALB resources ‚Äî but based on what‚Äôs shown, here are the **highest-risk infra pitfalls** to double-check:

#### 1) RDS EngineVersion `"15"` might fail

CloudFormation for RDS often expects an exact minor version (e.g., `"15.4"`). Sometimes `"15"` works, sometimes it doesn‚Äôt depending on API behavior.

**Safer MVP move:** remove `EngineVersion` entirely to use AWS default, or set it to a known valid one.

#### 2) Secrets as CloudFormation parameters

You pass secrets as CFN parameters (`GeminiApiKey`, `JwtSigningSecret`, etc.). Even with `NoEcho`, these values end up:

* in the stack metadata (hidden),
* and typically in the ECS TaskDefinition environment if you wire them that way.

**Better pattern (still MVP-friendly):**

* Store secrets in Secrets Manager
* Use ECS TaskDefinition `Secrets:` to inject them at runtime

This also avoids accidental exposure in task definition revisions.

#### 3) NAT Gateway cost

You explicitly said ‚Äúkeep NAT in MVP‚Äù ‚Äî fine, but be aware NAT will be one of the biggest baseline costs. If you want to cut cost later:

* Add VPC endpoints for ECR, S3, CloudWatch Logs
* Or use public subnets for tasks (not recommended for prod but sometimes acceptable for MVP)
