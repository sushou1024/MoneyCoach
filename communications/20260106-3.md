## P0 issues (high priority to fix before dev starts)

### 1) Naming collision: **Screen IDs “Sxx” collide with Strategy IDs “Sxx”**

Your screens are labeled **S00–S25**, and strategies are **S01–S25**. This will cause repeated confusion in:

* engineering tickets
* analytics naming
* backend payload fields (`strategy_id: "S05"` vs screen “S05 Quiz”)

**Fix**: rename screens to something like `SC00–SC25` (or `UI00–UI25`) and keep strategies as `S01–S25`.

---

### 2) Your example JSON has a hard inconsistency (will confuse devs)

In **Normalized Portfolio example**:

```json
"holdings": [{ "value_usd": 120000 }],
"net_worth_usd": 12450.0
```

That’s mathematically inconsistent and contradicts your “LLM doesn’t do math” principle (even though it’s just an example, it undermines trust).

**Fix**: make example numbers consistent everywhere (OCR example → normalized portfolio → preview report → paid report).

---

### 3) Handling unsupported screenshots (Futures/Options/Open positions) is not specified

PRD originally mentions users may upload Futures/Options screenshots, but MVP Gap excludes them. In real usage, people *will* upload:

* Binance Futures positions
* options legs
* margin/leveraged positions
* forex *positions* (not currency balances)

Right now you say “Holdings screenshots only (portfolio tables)” but you don’t define the behavior when a user uploads out-of-scope screenshots.

**You must define one**:

* **Option A (recommended for MVP)**: detect these as `unsupported_asset_view` and show: “Futures/Options not supported in v1.0 MVP. Please upload Spot/Wallet/Broker holdings.”
* **Option B**: parse but **exclude** from valuation and show them under `unpriced_holdings` with a “not supported” reason.

Without this, you risk:

* wrong net worth
* wrong risk score
* broken trust immediately

---

### 4) “Ambiguous symbol across domains” resolution is described but not **contracted**

You wrote:

> “If a symbol exists in multiple domains, require user confirmation in S10 before finalizing.”

But you didn’t define the **backend ↔ app handshake** for this.

You need an explicit API behavior like:

* OCR returns `ambiguities[]` with candidate interpretations
* app shows selector
* app sends `resolution[]` back
* backend persists resolution per user

Otherwise engineering won’t know how to implement “require user confirmation”.

---

### 5) Risk preference taxonomy mismatch vs strategy tech plan

Your onboarding captures:

* `risk_preference: Yield Seeker | Speculator`

But strategy formulas in the strategy doc use:

* `user_risk_level: conservative | moderate | aggressive`

You need a mapping, otherwise S01/S04/S05 parameter computation won’t be consistent.

**Fix**: define a deterministic mapping, e.g.

* Yield Seeker → conservative
* Speculator → aggressive
  …and decide where “moderate” comes from (experience/style/pain points), or collapse the strategy library to 2 tiers for MVP.

This affects:

* S01 stop loss %
* S04 layer sizing
* S05 DCA sizing/frequency

## P1 issues (important, but not blockers if you make quick decisions)

### 6) Pre-check “local/lightweight model” intent is not fully carried through

PRD emphasizes token-saving pre-check (“cat photos”, invalid uploads) **before** expensive OCR/LLM.

Your prototypes doc has a pre-check screen, but it’s unclear whether:

* pre-check happens **on-device** (cheap) before upload
* or after upload (costly)
* or via a smaller model before Gemini OCR

**Add a clear rule**:

* “On-device classifier” OR “server-side lightweight classifier” runs before Gemini
* if fails: don’t call Gemini; return INVALID_IMAGE early

---

### 7) “Value_display/value_usd” and valuation source-of-truth needs tightening

You have `value_display` from OCR + code-based valuation using providers. But you didn’t fully specify:

* When to trust screenshot value vs recomputed market price
* How to handle “stale screenshot” (user screenshot from last week)
* Whether preview/premium report always use current snapshot prices (PRD implies yes)

You *did* say CMC quotes are source of truth for valuation ✅, but your schemas still carry both.

**Recommendation** (contract clarity):

* Always store **both**:

  * `value_from_screenshot` (optional)
  * `value_usd_priced` (computed, source-of-truth)
* And include a field like:

  * `pricing_source: "CMC" | "MARKETSTACK" | "USER_PROVIDED"`

---

### 8) Dedupe rule may create false positives without an override

Your dedupe rule (same platform_guess + row count + exact total match) is reasonable, but two different screenshots can still match totals (especially if UI rounds totals).

**Add**:

* A “Review duplicates” UI or a “Restore excluded screenshot” toggle
* Or store duplicates but mark as `excluded_reason: "duplicate_suspected"` and allow user override

---

### 9) Forex “in scope” is underspecified (balances vs positions)

You say Forex is supported, but also say:

* no forex signals in MVP
* no forex chart in MVP

That’s okay, but you should define **what forex means**:

* **FX balances** (e.g., EUR cash held at broker) — easy
* **FX trading positions** (e.g., long EURUSD with leverage) — complex and probably out-of-scope

Right now it’s ambiguous and will break OCR normalization.

---

### 10) Preview/Paid report LLM settings are missing

PRD explicitly references temperature guidance (OCR=0, analysis low but not 0). Your prototypes doc sets OCR temperature but not:

* preview LLM temperature
* paid report LLM temperature
* max tokens and JSON enforcement approach

**Add**:

* preview: temp=0.2 (or 0.1)
* paid: temp=0.2
* strict JSON schema validation + retry policy (e.g., 2 retries then fail)

---

### 11) Count + mapping invariants for risks are implied but not explicit

PRD uses “3 key risks” and paid report must justify the preview risks.

You should explicitly enforce:

* preview returns **exactly 3** `identified_risks`
* paid report `risk_insights` must be a **1:1 expansion** of those risk ids/types
  (e.g., `risk_01` expands into the first paid insight)

Right now it’s described conceptually but not enforced as a contract.

## Strategy-spec alignment issues (MVP strategy details vs your deterministic rules)

### 12) S05 deterministic sizing rule deviates from the strategy doc

Strategy doc’s S05 amount uses `user_monthly_income` and `user_savings_rate`, but your onboarding does not collect them, so your doc proposes:

> `base_amount = min(idle_cash_usd * 0.10, net_worth_usd * 0.02)` etc.

This is fine for MVP, but it’s a **spec divergence** that should be recorded as an intentional deviation (similar to your MVP Gap list), otherwise you’ll fight later about “which formula is correct”.

**Action**: add a short “MVP Simplification” note under S05:

* “Income-based DCA sizing deferred; MVP uses idle cash + net worth heuristic.”

---

### 13) S04 trigger uses “holding time > 30 days” in strategy doc, but MVP data may not have it

Unless screenshots expose holding duration (often they don’t), you can’t reliably enforce “holding_time > 30 days”.

You should decide:

* Drop that condition for MVP, or
* Interpret “holding time” as “time since first scan snapshot”, which biases behavior

Right now it’s not addressed, so S04’s “must satisfy” constraints are not implementable.

---

### 14) You require pnl-dependent strategies but don’t define how to compute PnL deterministically when fields are missing

You say:

> If pnl_percent missing, treat profit/loss-dependent strategies as not applicable.

That is safe, but reduces the chance of recommending S02/S03/S04.

PRD suggests prompting user to add avg cost to unlock recovery strategies — you included that ✅.
But you should also specify deterministic fallback PnL calculation when possible:

* If `avg_price` present and `current_price` known → compute `pnl_percent` in code.
* If only screenshot shows “unrealized PnL” but not avg price → you can still compute `pnl_percent` using `value_usd` and implied cost (sometimes not possible).

Right now this is partially described but not fully specified.
