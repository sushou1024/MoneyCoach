## 2) Notable divergences from PRD (not “wrong,” but should be explicitly recorded)

These are **not contradictions with the MVP Gap list**, but they *do* differ from what the PRD “implies” or states in non-gap sections. I’d recommend either (a) add them to the MVP Gap list, or (b) adjust PRD text to match your implementation spec so future readers don’t get confused.

### 2.1 Health score ownership: PRD implies “LLM judges”; prototypes make backend the anchor

* **PRD (free prompt)**: suggests LLM assigns `health_score` (institutional judgment).
* **Prototypes**: backend computes baseline via a deterministic formula and **clamps LLM adjustments within ±5**.

This is a *good engineering choice* (stability + debuggability), but it’s a conceptual divergence: it changes “LLM is the score source-of-truth” into “backend is source-of-truth; LLM is copy/justification with bounded adjustment.”

**Recommendation**: Document this as an explicit MVP decision:

> “Health/volatility scores are backend-anchored with bounded LLM adjustment; persisted values are immutable across preview→paid.”

### 2.2 OCR prompt text in PRD conflicts with MVP FX handling

PRD contains an OCR prompt that explicitly says:

* “estimate USD value based on standard exchange rates … mark currency_converted true”

But your prototypes correctly implement MVP Gap #9:

* OCR returns `display_currency` + raw value; backend performs FX conversion.

**Recommendation**: Update the PRD’s included OCR prompt section to avoid future reintroduction of FX conversion in the OCR layer.

### 2.3 “Free users Insights” is now a product decision (not specified in PRD)

You’ve decided:

* Free users can fetch in-app Insights feed, but **push notifications require paid entitlement**.

PRD doesn’t explicitly define this entitlement split for Insights (it’s implied but not spelled out).
Not a conflict, but worth making explicit in PRD or in an “Entitlement policy” section.

### 2.4 “Paid reports remain read-only after lapse”

You’ve chosen:

* Reports remain viewable, but uploads and paid generation are restricted after lapse.

Again: good and common, but PRD doesn’t state it.

## 3) Internal inconsistencies and “spec friction” between your prototypes doc and backend spec

These are small but worth fixing because they’ll cause implementation confusion.

### 3.1 Dedupe timing / fingerprint: flow step order is inconsistent

* In prototypes doc you say: **compute fingerprints immediately after OCR so duplicates are shown in `needs_review`**.
* In backend spec “Core Data Flow (Holdings)” step list, fingerprint/dedupe appears after review (“normalize symbols, dedupe”).
* But later the backend spec also says: “Dedupe runs after OCR and before returning needs_review.”

So the *step numbering* in the flow section is inconsistent with your later dedupe rules.

**Recommendation**: pick one canonical truth (your later dedupe rules are better), and update the “Core Data Flow” numbering to match:

* OCR → fingerprint/dedupe (best-effort) → draft pricing → needs_review → review → normalize+snapshot.

### 3.2 Pre-check contract: `is_portfolio_screenshot` vs `reason_code=UNSUPPORTED_VIEW`

You define:

* Pass rule: `is_portfolio_screenshot=true AND confidence>=0.70`
* But also: if `reason_code=UNSUPPORTED_VIEW`, skip OCR and mark ignored_unsupported.

That implies `UNSUPPORTED_VIEW` must override pass/fail, but the contract doesn’t explicitly say precedence.

**Recommendation**: Make precedence explicit:

* If `reason_code == UNSUPPORTED_VIEW`, treat as “portfolio-like but unsupported,” always skip OCR regardless of confidence.

### 3.3 “Value_from_screenshot” semantics: reference-only vs user-provided valuation is unclear

You say:

* `value_from_screenshot` is reference only
* `value_usd_priced` is recomputed from amount + snapshot price
* unpriced holdings are excluded from net worth unless user-provided

…but you don’t fully define **how a user can “user-provide” a value** for an unpriced asset in SC10 (or whether you allow it in MVP).

Your data model includes `valuation_status: user_provided` and `pricing_source: USER_PROVIDED`, but the SC10 UX says screenshot value edits are “reference only.”

This is a genuine **under-spec**.

**Recommendation (pick one MVP policy):**

* **Option A (simplest)**: No user-provided valuations in MVP. If symbol can’t be priced/validated, it’s excluded, period. Then remove `user_provided` from MVP data dictionary.
* **Option B (more user-friendly)**: Allow SC10 to mark an item as “manual value” when pricing fails; store `value_usd_priced = user_input_usd`, set `valuation_status=user_provided`, clearly label in UI and exclude from strategy parameter logic.

Right now your docs imply both.

### 3.4 Strategy selection logic is under-specified (parameters are specified)

You did an excellent job specifying **parameter computation** for S01–S05, but the **strategy selection / plan construction** is only partially stated.

Example: You say:

* if cost basis unknown, skip S02/S03/S04, allow S01 and S05
* stablecoin idle fallback: recommend cash allocation plan or S05

But you do not define a deterministic selection policy like:

* when do we recommend S01 vs S03 for a profitable asset?
* do we always recommend S01 for every holding (since it’s “普适推荐”)?
* do we cap number of plan cards? (PRD implies multiple cards by severity)
* do we prioritize portfolio-level issues vs per-asset issues?

**Recommendation**: Add a “MVP Strategy Plan Constructor” section with deterministic rules:

* max plan count (e.g., 3)
* per-asset eligibility gates
* severity ranking
* portfolio-level cash inefficiency handling
* “always include S01?” policy (or include only for top N risk holdings)

### 3.5 Asset identity collisions: symbol-only storage is risky (CMC & Marketstack both have collisions)

You rely heavily on symbol validation:

* crypto via CMC `/map`
* stocks via Marketstack `/tickers`

In reality:

* **CMC symbol collisions** are common (multiple tokens share the same symbol).
* **Stock symbol collisions** across exchanges/regions are common.

Your current schema stores `symbol` but not canonical IDs (CMC id, marketstack ticker id).

That is an ambiguity risk:

* You may price the wrong token even if user chose “ABC coin” vs “ABC stock,” because “ABC coin” itself may map to multiple CMC ids.

**Recommendation** (strong):

* Store canonical identifiers:

  * `crypto_cmc_id` (or generic `asset_provider_id`)
  * `marketstack_ticker_id` or `exchange_mic + symbol`
* Make pricing snapshotted per canonical id, not symbol string.

### Potential CloudFormation footguns (worth addressing)

These aren’t “doc inconsistencies,” but they’re real “deployment surprises”:

1. **RDS EngineVersion = "15"**
   In many regions, RDS Postgres expects a full minor version (e.g., `15.4`). `"15"` *may* work or may fail depending on AWS behavior/region.
   **Recommendation**: either remove EngineVersion (let AWS default), or parameterize it.

2. **Single NAT Gateway** is fine for MVP, but it’s a single point of failure.
   Probably acceptable; just note it.

3. **Redis CacheCluster (not ReplicationGroup)** is fine for MVP, no HA.
   Acceptable; just document that Redis is not HA.

4. **Uploads bucket DeletionPolicy Retain**: good for safety but can surprise devs (stack delete won’t remove bucket).
   If intentional, fine—document it.

### GitHub Actions workflow consistency

* Builds and pushes image to ECR ✅
* Deploys CF stack with parameter overrides ✅
* Passes env vars described in the backend spec ✅

One thing to note: your workflow always passes a long list of parameters, and many of them may be empty strings if GitHub Vars aren’t set. CloudFormation will happily accept empty strings; your service will start but payment/auth integrations may break at runtime unless code handles missing configs gracefully.

**Recommendation**: Add a startup “config validation” step (see env var section below).

### 6.2 Where env vars are confusing today

Your docs say “Payments env vars required when channel enabled,” but your CloudFormation and CI pass them *unconditionally* (even if blank). That creates ambiguity:

* “Are Stripe vars required for iOS-only launch?”
* “Can I deploy backend without setting Play Billing keys?”
* “What breaks if a value is empty?”

Right now the answer is: **unclear** unless the code explicitly handles empty values.

### 6.3 Suggested classification: required vs conditional

#### Always required for *any* environment that serves traffic

These are truly core:

* `JWT_SIGNING_SECRET`
* `DATABASE_URL`
* `REDIS_URL`
* `OBJECT_STORAGE_BUCKET`
* `OBJECT_STORAGE_REGION`
* `GEMINI_API_KEY`
* `CMC_PRO_API_KEY`
* `MARKETSTACK_ACCESS_KEY`
* `OPEN_EXCHANGE_APP_ID`
* `BINANCE_API_BASE_URL`

Likely also required if you ship email auth:

* `RESEND_API_KEY`
* `RESEND_FROM_EMAIL`

#### Conditionally required (feature flags)

These should be required *only if the feature is enabled*:

**Google OAuth login**

* `GOOGLE_ALLOWED_CLIENT_IDS`
  If you allow Google sign-in, this should be **non-empty** and enforced; otherwise you risk accepting tokens minted for other apps.

**Stripe payments**

* `STRIPE_SECRET_KEY`
* `STRIPE_WEBHOOK_SECRET`
* `STRIPE_PRICE_ID_WEEKLY`
* `STRIPE_PRICE_ID_YEARLY`

**Apple IAP**

* `APPSTORE_SHARED_SECRET`
* `APPLE_IOS_BUNDLE_ID`
* `APPLE_IAP_PRODUCT_ID_WEEKLY`
* `APPLE_IAP_PRODUCT_ID_YEARLY`

**Google Play billing**

* `GOOGLE_PLAY_PACKAGE_NAME`
* `GOOGLE_PLAY_SUBSCRIPTION_ID`
* `GOOGLE_PLAY_BASE_PLAN_ID_WEEKLY`
* `GOOGLE_PLAY_BASE_PLAN_ID_YEARLY`
* `GOOGLE_PLAY_SERVICE_ACCOUNT_JSON_BASE64`

**Apple web auth (PWA only)**

* `APPLE_SERVICE_ID`
* `APPLE_REDIRECT_URI`
* `APPLE_POST_AUTH_REDIRECT_URI`
* `APPLE_TEAM_ID`
* `APPLE_KEY_ID`
* `APPLE_PRIVATE_KEY_P8_BASE64`

#### Optional/infra knobs (good to keep optional)

* `PORT` (but you already set it)
* `UploadsCorsAllowedOrigins` / `UPLOADS_CORS_ALLOWED_ORIGINS`
* `ACM_CERTIFICATE_ARN` (optional HTTPS)

### 6.4 “Optional env vars confuse people” — practical fix

You can eliminate most confusion without changing infra by adding:

1. **Explicit feature flags**, e.g.

* `ENABLE_STRIPE_PAYMENTS=true|false`
* `ENABLE_APPLE_IAP=true|false`
* `ENABLE_GOOGLE_PLAY=true|false`
* `ENABLE_APPLE_WEB_AUTH=true|false`
* `ENABLE_EMAIL_AUTH=true|false`

2. **Startup validation**:

* If a feature flag is true, validate its required env vars are present (non-empty) and crash fast with a clear error like:

  * “ENABLE_STRIPE_PAYMENTS=true but STRIPE_SECRET_KEY is missing.”

3. **Docs alignment**:

* Update the backend spec “Required Environment Variables” section to list:

  * **Always required**
  * **Required if ENABLE_X=true**

This is the single biggest way to stop env var confusion.
