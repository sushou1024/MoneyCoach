## 1) Holdings OCR prompt (“Quanta Vision Parser”) — contract mismatches

### What the PRD prompt currently asks for

Key parts in PRD OCR prompt:

* “Analyze the provided image(s).”
* Extract:

  * Symbol
  * Amount
  * **USD Value**
  * Optional: pnl_percent, avg_price
* “Currency Normalization: estimate USD based on exchange rates”
* Output schema (single object):

  ```json
  {
    "status": "success",
    "platform_guess": "Binance",
    "assets": [
      { "symbol": "BTC", "amount": 1.25, "value_usd": 122500.00, "pnl_percent": 15.2, "avg_price": 89000.00 }
    ]
  }
  ```

### What your MVP prototypes/backend spec actually need

Your backend/prototypes explicitly enforce MVP deviations:

* **No FX conversion in OCR** (backend does it)
* OCR should output **display_currency + value_from_screenshot**, not `value_usd`
* You want **per‑image status** handling: success / ignored_invalid / ignored_blurry / ignored_timeout / ignored_unsupported, and you want to return those per image for SC10 review and for batch‑level error precedence.
* You have a whole dedupe + “unsupported view” pipeline that requires knowing which image produced which rows.

### Hard consistency issues (must fix)

These are *data‑form mismatches* that will break or weaken the workflow if you use the PRD OCR prompt “as is”:

#### Issue A — Multi‑image ambiguity

PRD prompt says “image(s)”, but output schema is a **single** `platform_guess` and a **single** `assets[]`.

* In your flow, users upload **1–15 images** and they can be from multiple platforms (Binance + MetaMask + broker).
* If you call OCR once with multiple images, the model cannot cleanly produce per‑image platform + per‑image assets with the current schema.
* If you call OCR per image (which your backend spec implies), the PRD exception handling (“If all uploaded images fail…”) is not applicable and the “image(s)” language is misleading.

**Result:** your backend won’t be able to reliably attach rows to `image_id`, nor surface per‑image errors, nor dedupe correctly.

✅ Fix: make the OCR prompt explicitly **single‑image** (recommended), or change output schema to `images[]` with `image_id` and per‑image assets. Your prototypes strongly imply single‑image calls → align the PRD prompt to that.

#### Issue B — FX conversion + `value_usd` conflicts with your MVP pipeline

PRD OCR prompt explicitly instructs:

* estimate USD value
* output `value_usd`

But your MVP pipeline says:

* OCR returns `display_currency + value_from_screenshot`
* backend converts using OpenExchangeRates and sets `currency_converted`

If you keep PRD prompt as-is, you’ll have:

* **two USD valuations** (model estimate + backend priced)
* hard‑to‑debug drift (“why does OCR value_usd ≠ priced USD?”)
* incentive for hallucination when screenshots don’t show per‑asset value

✅ Fix: remove FX conversion from OCR prompt and rename fields to `value_from_screenshot` + `display_currency` (optional, only if visible).

#### Issue C — Missing `asset_type` in PRD OCR schema

Your downstream normalization logic and SC10 ambiguity resolution **needs** asset domain:

* `crypto | stock | forex` (even if “tentative”)
* plus later canonicalization via `asset_key`

The PRD OCR output does not include `asset_type`.

You *can* infer from platform_guess, but:

* broker screenshots can include FX cash + stocks
* wallets/exchanges can show USD as “cash-like” text labels
* symbol collisions exist (“ABC”, “MATIC” vs equities etc)

✅ Fix: include `asset_type` in the OCR output per row (or explicitly mark unknown + let backend disambiguate, but then the prompt should output a list of ambiguities candidates; your current PRD OCR prompt doesn’t).

#### Issue D — PRD JSON examples contain comments (`// Optional`) → invalid JSON

In PRD prompts, the “strict JSON” examples contain lines like:

```json
"pnl_percent": 15.2, // Optional
```

This is **not valid JSON** and will cause:

* strict schema validation failures
* retries / increased latency
* model “learning” that comments are acceptable

✅ Fix: remove comments from all JSON examples in prompts. If you want optional fields, describe them in natural language, not JSON comments.

### “Backend has everything necessary to call OCR LLM?”

Yes, but only if you align the prompt to your real pipeline:

* You already have `image_id`, and you already know the purpose = holdings
* You have pre‑check outputs
* You have dedupe and error precedence logic
* You have SC10 review needs

**But the current PRD OCR prompt output is not sufficient** to support:

* per‑image status
* per‑image platform mapping
* dedupe visibility in SC10
* downstream asset_type ambiguity workflow

1. **LLM output schema** (no numeric parameters)

Right now, your written docs partially mix them.

**Recommendation:** explicitly document the merge step and define the LLM schema separately.

## 5) Concrete prompt rewrites that match your MVP pipeline

### 5.1 MVP Holdings OCR system prompt (single image, per-image status)

Key goals:

* no FX conversion
* no USD estimation
* per-image status
* valid JSON only
* include `asset_type`

```text
You are the "Quanta Vision Parser (MVP)".

Your ONLY job is to extract holdings rows from ONE screenshot and return a strict JSON object.

This screenshot may come from:
- Crypto exchanges (spot balances only), wallets
- Stock brokerages (equities/ETFs holdings)
- Forex cash balances (cash only)

MVP Restrictions:
- Futures, options, margin, leveraged positions are NOT supported.
- Do NOT perform FX conversion.
- Do NOT estimate USD values.
- Only return values that are explicitly visible on the screenshot.

PII:
- Do NOT output any emails, phone numbers, account IDs, addresses, or names.

TASK:
1) Decide the image status:
   - "success": a holdings/asset list is visible and readable
   - "ignored_invalid": not a portfolio/holdings screenshot
   - "ignored_unsupported": futures/options/margin/positions screen
   - "ignored_blurry": too blurry to extract reliably
2) If success, extract each visible asset row.

EXTRACTION RULES:
- symbol_raw: exactly as shown (e.g., "Bitcoin", "Tether", "BRK.B")
- symbol: best-effort normalized ticker (e.g., Bitcoin -> BTC; Tether -> USDT)
- asset_type: one of "crypto" | "stock" | "forex"
- amount: numeric quantity / shares / cash amount
- value_from_screenshot: numeric value shown for that row ONLY if visible; otherwise null
- display_currency: currency code/symbol for value_from_screenshot if visible (e.g., USD, USDT, EUR, ¥); otherwise null
- avg_price: only if explicitly shown; otherwise null
- pnl_percent: only if explicitly shown; otherwise null

Output MUST be valid JSON only (no markdown, no comments).

OUTPUT SCHEMA:
{
  "status": "success|ignored_invalid|ignored_unsupported|ignored_blurry",
  "platform_guess": "Binance|OKX|MetaMask|Robinhood|IBKR|Unknown",
  "assets": [
    {
      "symbol_raw": "string",
      "symbol": "string",
      "asset_type": "crypto|stock|forex",
      "amount": 0.0,
      "value_from_screenshot": 0.0,
      "display_currency": "string",
      "avg_price": 0.0,
      "pnl_percent": 0.0
    }
  ]
}
```

This prompt now matches your backend expectation:

* backend attaches `image_id`
* backend computes confidence and FX conversion
* backend can map platform_guess → platform_category
* backend can handle ignored statuses cleanly
