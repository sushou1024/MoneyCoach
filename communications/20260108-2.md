## 1) MVP simplifications you can lift now (no new data sources, no invariant changes)

These are “safe lifts” because they **don’t require new providers** and can stay within **S01–S05 + MarketAlpha strategy-free** rules.

### 1.1 Lift: Make S01 truly “institutional” using its own dynamic rules (and S13 behavior)

Your strategy library S01 already includes “dynamic adjustment rules” (profit 5% → move to breakeven, etc.). Your current prototypes mostly treat S01 as a static stop-loss.

**What to lift:**

* Implement S01 “dynamic adjustment rules” as *runtime plan evolution*:

  * profit ≥ 5% → move stop to breakeven (avg_price)
  * profit ≥ 10% → move stop to lock some profit
  * profit ≥ 20% → (optional) “upgrade to S03” (allowed, since S03 is in MVP allowlist)

**Why this is liftable now:**

* Requires only **current price** + **avg_price** (or derived avg_price) + **pnl%**.
* Uses existing infrastructure you already designed:

  * `plan_states` (you already have `peak_price_since_activation`, `activated_at`, etc.)
  * Insights Portfolio Watch triggers

**Implementation shape (fits your architecture):**

* Add to `plan_states.state_json` for S01:

  * `current_stop_loss_price`
  * `stage` (e.g., `initial | breakeven | profit_lock`)
  * `last_adjusted_at`
* Add a scheduled job (can piggyback `insights_refresh`) that:

  * recomputes whether the next dynamic rule should apply
  * updates `current_stop_loss_price`
  * optionally emits an insight like: “Stop moved up to protect capital”

**Guardrail:**

* Do not mutate the original “report parameters” that were snapshot-locked; treat this as **runtime plan state** that lives in Insights (dynamic), while the report remains a snapshot.

This lifts a big “MVP simplicity” without adding new strategies.

## 6) “Meticulous” spec issues I recommend fixing before lifting scope

These are small inconsistencies in your prototype docs that will cause confusion or bugs once you expand.

### 6.1 OCR “confidence” field contradiction

In your prototype doc you say:

* “confidence is backend-computed; LLM must not output confidence.”
  …but earlier you also list `confidence` in the “Per-image response schema (required)”.

**Fix:**

* Make it explicit:

  * **LLM OCR schema**: no `confidence`
  * **Backend API response**: includes `confidence`

### 6.2 Preview/Paid token limits inconsistent across docs

Prototype: preview/payed `max_output_tokens 4096`
Backend spec: preview 1200, paid 2400

**Fix:** pick one set and document it in one place as source-of-truth.

### 6.3 “warnings[]” ownership is unclear

If the model must not output PII, but you want `warnings=["PII_REDACTED"]`, the backend won’t know PII existed unless:

* model outputs a boolean “pii_detected”
* or you run a separate OCR/text detector (which you’re not)

**Fix options:**

* Option A: make `warnings[]` model-generated but restricted to enums and no raw PII
* Option B: drop PII warnings in MVP, keep only “avoid sensitive info” UX copy

### 6.4 Strategy/report immutability vs plan evolution

You already handle “delta updates don’t recompute the optimization plan” (good for immutability).

But if you lift dynamic behaviors (S01 dynamic stops, S04 post-layer protection), you must document clearly:

* **Report is a snapshot**
* **Plan state is live**
* Insights uses plan state, not report parameters

You already started this (“Report Snapshot based on scan date”), but it should become a first-class rule.

### 6.5 Market Alpha “strategy-free” rule must stay true in schemas

You did the right thing: `market_alpha.strategy_id=null`.
If you ever add “Create a plan from this signal” later, keep the feed item strategy-free; create a separate action endpoint to instantiate a plan.
