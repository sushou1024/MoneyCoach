## 2) The biggest **logic consistency risk** in your system right now: `report_id` vs `calculation_id`

You currently have a **semantic ambiguity** across:

* your prototype “Identifier Lifecycle”
* your endpoint shapes
* your DB schema

### What your docs *currently imply*

* Preview report returns **both** `report_id` and `calculation_id`.
* Paid upgrade endpoint is `POST /reports/{calculation_id}/paid`, returning a `report_id`.
* Examples show preview and paid sharing the same `report_id` (`rpt_888`), i.e. “report_id is stable, calculation_id is stable”.

### Why your DB schema is at risk

Your `reports` table has:

* `id` (pk)
* `calculation_id`
* `report_type (preview|paid)`
* `payload`

That schema strongly suggests either:

1. **two rows** per calculation (preview and paid), OR
2. **one row** that gets “upgraded” from preview→paid (but then `report_type` becomes “paid” and you lose “preview record-ness”).

But your examples show the same `report_id` for preview and paid, which conflicts with “two rows” (because `id` must be unique).

### Pick one model and make everything match

Here are the clean options (choose one and align docs + schema + endpoints accordingly):

#### Option A (recommended): `calculation_id` is the true stable ID; `report_id` is optional

* Treat the “analysis result” as the primary entity: **calculation**.
* Preview and paid are **tiers** of the same calculation.
* Use **one row** keyed by `calculation_id`.
* Store `preview_payload` and `paid_payload` separately (or `payload` + `paid_payload`).
* `report_id` becomes unnecessary, or it becomes a derived alias of `calculation_id`.

**Schema shape (example):**

* `calculations` table:

  * `calculation_id (pk)`
  * `portfolio_snapshot_id`
  * `preview_payload jsonb`
  * `paid_payload jsonb null`
  * `status_preview`, `status_paid`
  * `created_at`

**Endpoint simplification:**

* `GET /reports/preview/{calculation_id}` reads preview_payload
* `POST /reports/{calculation_id}/paid` writes paid_payload
* `GET /reports/{calculation_id}` returns paid if exists else preview

This model matches your “consistency is law” logic and avoids “two IDs for one thing”.

#### Option B: `report_id` stable, upgrades in-place

* Create one `reports` row as preview.
* Paid upgrade **updates the same row** and flips `report_type` to `paid`.
* Preview endpoint just returns a subset view.

This works but:

* you lose explicit distinction of “preview object vs paid object”
* you must be careful with audit trails (what did the user see pre-upgrade?)

#### Option C: two report rows (preview_id, paid_id) + a parent link

* Keep preview row immutable.
* Create paid row with `parent_report_id = preview_report_id`.
* UI and backend always “resolve to paid if exists”.

This is clean for audit, but it adds mapping complexity.

**Why I’m calling this out as “biggest risk”:**
If you don’t lock this now, you’ll eventually get one of:

* duplicated “My Reports” entries
* inability to fetch paid report by preview id
* messy migrations when analytics depends on stable identifiers

### ✅ What you did right

* Parameter generation is deterministic and versioned by the backend.
* You correctly gate S02/S03/S04 by cost basis availability.

### Two edge cases to handle explicitly (otherwise you’ll get weird user complaints)

1. **avg_price present, pnl_percent present but inconsistent**
   In SC10 user edits, they can set both; if inconsistent with snapshot price, what wins?

   * Recommendation: treat `avg_price` as source-of-truth and recompute `pnl_percent` in code at snapshot time.
   * Keep original OCR pnl in audit fields if needed, but don’t propagate inconsistency.

2. **S02 “idle_cash_usd” dominated by user_provided valuations**
   You exclude user_provided from strategies (good), but your docs say idle_cash_usd sums stablecoins + cash-like holdings. If a user manually values something as “cash”, you should make sure it doesn’t accidentally inflate idle_cash and trigger martingale sizing.

   * Recommendation: define `idle_cash_usd_strategies = sum(priced cash-like holdings only)`
     and use that for S02/S05.

## 7) CloudFormation template: key issues to address based on what you shared

You shared the stack up to the S3 CORS section (truncated), but even from that there are some important things.

### P0: Add S3 lifecycle retention to match your backend spec (30 days)

Your backend spec says:

> retention 30 days via S3 lifecycle

But in the snippet shown, I only see encryption + public access blocks, not lifecycle (maybe it’s below the truncation; if not, add it).

**Add this to `UploadsBucket.Properties`:**

```yaml
LifecycleConfiguration:
  Rules:
    - Id: ExpireUploadsAfter30Days
      Status: Enabled
      ExpirationInDays: 30
      AbortIncompleteMultipartUpload:
        DaysAfterInitiation: 7
```

### P0: Fix/complete S3 CORS (AllowedHeaders/ExposeHeaders)

For future PWA direct uploads, you almost always want `AllowedHeaders: ["*"]` and at least `ExposeHeaders: ["ETag"]`.

Example:

```yaml
CorsConfiguration:
  CorsRules:
    - AllowedMethods: [GET, PUT, HEAD]
      AllowedOrigins: !Split [",", !Ref UploadsCorsAllowedOrigins]
      AllowedHeaders: ["*"]
      ExposedHeaders: ["ETag"]
      MaxAge: 3000
```

### P0/P1: Redis “CacheCluster” is not ideal for security

`AWS::ElastiCache::CacheCluster` (Redis) is fine for MVP, but:

* no auth token (unless you use replication group features)
* in-transit encryption usually requires replication group

If you care about protecting user financial signals and plan state (you should), strongly consider switching to:

* `AWS::ElastiCache::ReplicationGroup` with:

  * `TransitEncryptionEnabled: true`
  * `AtRestEncryptionEnabled: true`
  * `AuthToken` (store in Secrets Manager)

If you *don’t* switch now, at least ensure:

* Redis SG is strictly limited (you already did)
* Redis is in private subnets only (you did)

### P1: Secrets handling via CloudFormation parameters

You’re passing API keys as stack parameters (with `NoEcho`). That works, but it still means:

* secrets are “configuration” rather than “secret material” from an AWS perspective
* operationally, rotating keys is noisier

Better pattern:

* store all third-party keys in **Secrets Manager**
* reference them from ECS TaskDefinition via `secrets:` rather than `environment:`

You already created `DatabaseCredentials` as a Secret — doing the same for provider keys would be consistent and safer.

### P1: Single NAT gateway is a single point of failure

For MVP it’s acceptable, but it’s worth explicitly noting:

* NAT is in one AZ; if that AZ has issues, your private subnets lose outbound access (LLM + market data calls die).
  A “production hardening” follow-up would be dual NATs, one per AZ.
