## 2) P0 issues: hard inconsistencies you should fix (or explicitly decide)

These are the ones that will otherwise cause engineers to build different interpretations.

### P0.1 SC10 “priced USD values” vs backend flow “pricing only after SC10 confirm”

**In prototypes UI spec (SC10)** you want to show per-row:

* `Value (USD, priced)` and allow edits that “recalculate value_usd_priced using current market prices”.

But in your pipeline description and backend spec:

* After OCR, batch returns `status=needs_review` with `ocr_assets` **without priced values**.
* You also state: “Normalization, pricing, preview generation start only after SC10 confirmation.”

**This is currently inconsistent.** You can’t show `value_usd_priced` at SC10 if pricing hasn’t happened yet.

**Fix options (choose one and make it explicit):**

1. **Option A (recommended): Draft pricing before confirm**

   * After OCR completes (still `needs_review`), backend performs:

     * symbol validation (or at least “best effort”)
     * a **draft quote fetch** (not yet “snapshot locked”)
   * Return `value_usd_priced_draft` + `price_as_of` in `needs_review` response.
   * After user confirms, backend creates a **locked `market_data_snapshot`** and recomputes final pricing for the report.
2. **Option B: SC10 shows only screenshot-native values**

   * SC10 shows amount + screenshot value + currency.
   * Only after confirm do you show USD-priced values in SC12 preview.
   * Downside: SC10 becomes weaker for user trust/corrections.
3. **Option C: Add a dedicated endpoint**

   * `POST /v1/upload-batches/{id}/price-draft` returns priced draft rows for review.

If you keep SC10 as currently written, you’ll want **Option A**.

### P0.2 “All timestamps are UTC” vs your own examples storing local offsets

Backend spec says: **“All timestamps: ISO-8601 UTC.”**
But your examples and rules say:

* `next_execution_at` is stored in user local timezone (and example uses `-05:00` offset).

That’s a spec contradiction.

**Fix options:**

* **Option A (recommended): store and return UTC, plus timezone**

  * Store `next_execution_at_utc` (Z)
  * Store `user_timezone` separately (IANA string)
  * App renders “09:00 local” by converting UTC -> local.
* **Option B: relax the API convention**

  * Change the convention to “ISO-8601 timestamps with timezone offsets allowed.”
  * Then be consistent everywhere.

### P0.4 Strategy plan schema can’t uniquely represent “same strategy on multiple assets”

Right now, your report payload and DB models imply:

* `optimization_plan[]` items have `strategy_id` and `parameters`, but **no asset key**.
* Endpoint `GET /v1/reports/{report_id}/strategy/{strategy_id}` is ambiguous if you have:

  * S01 stop loss for BTC AND S01 stop loss for ETH
  * S05 DCA for BTC AND S05 DCA for SPY, etc.

**You need a strategy instance identity.**
Recommended pattern:

* In the report: each plan item has

  * `plan_id` (or `strategy_instance_id`)
  * `strategy_id` (S01..S05)
  * `asset_type`
  * `symbol`
  * `parameters` …

Then your detail endpoint becomes:

* `GET /v1/reports/{report_id}/plans/{plan_id}`
  or
* `GET /v1/reports/{report_id}/strategy/{strategy_id}?symbol=BTC&asset_type=crypto`

Same applies to `report_strategies` table: add `symbol` (and `asset_type`) columns, or store them inside `parameters` and enforce them.

### P0.5 Paid conversion flow is under-specified (receipt -> entitlement -> paid report)

Your UI flow says: “Paywall purchase success -> SC14 Processing (Paid).”
But the backend requires a valid entitlement for `POST /v1/reports/{calculation_id}/paid`.

You need to specify, per platform:

* **iOS (IAP)**:

  1. App completes purchase
  2. App sends receipt to backend `POST /v1/billing/receipt/ios`
  3. Backend marks entitlement active
  4. App calls `POST /v1/reports/{calculation_id}/paid`

* **Android (Play Billing)**:

  1. Purchase token -> backend `POST /v1/billing/receipt/android`
  2. Entitlement active -> generate paid report

* **Stripe (Android/PWA)**:

  1. App opens Stripe Checkout session URL
  2. Stripe webhook updates entitlement
  3. App polls `GET /v1/billing/entitlement` until active
  4. App calls `POST /v1/reports/{calculation_id}/paid`

Right now the “glue” steps aren’t written, so different devs will invent different flows.

## 3) P1 issues: ambiguity / under-specifications that will likely bite later

### P1.1 “Health Score” source of truth is unclear (LLM vs backend rubric)

PRD framing: health_score is “LLM judgment” (institutional nurse).
Your prototypes introduce:

* backend computes feature_vector and baseline scores
* LLM may adjust within +/-5
* backend clamps

That’s a *good engineering move*, but right now it’s underspecified in two ways:

1. **What is the actual deterministic rubric?** (mapping from feature_vector -> baseline score)
2. **Who is the source of truth?**

   * If backend clamps, then backend is source of truth and LLM is advisory.

**Recommendation:** explicitly define:

* the deterministic baseline formula (even if simple)
* final score = clamp(LLM_score, baseline±5) OR simply baseline and LLM doesn’t output scores at all.

This matters because PRD also says: “once generated it cannot change” (you already preserve that) — but you still need consistent generation.

    Many native setups instead rely on `google-services.json`, so this may not be needed. Just be aware.

**Best practice to reduce optional confusion:**

* Prefer to fetch plan/product IDs from `GET /v1/billing/plans` at runtime.
  If you do that, the mobile env can shrink to basically:

  * `API_BASE_URL`
  * Google client IDs
  * Stripe publishable key (Android/PWA only)

#### 5.2.1 Naming inconsistencies to fix (P0 for DX)

Your backend spec earlier listed variables like:

* `CMC_API_KEY` (but env uses `CMC_PRO_API_KEY`)
* `MARKETSTACK_API_KEY` (but env uses `MARKETSTACK_ACCESS_KEY`)
* `OER_APP_ID` (but env uses `OPEN_EXCHANGE_APP_ID`)

* `APPLE_APP_BUNDLE_ID` (but env uses `APPLE_IOS_BUNDLE_ID`)

Pick one naming scheme and make:

* docs

* `.env.example`
  all match exactly.

#### 5.2.2 Missing env vars you almost certainly need (non-DB/Redis)

Even excluding DB/Redis, your backend spec itself requires things not present in `.env.example`:

* **JWT signing secret** (required for auth):

  * `JWT_SIGNING_SECRET=...`

* **Object storage config** (required because you explicitly use pre-signed uploads):

  * `OBJECT_STORAGE_BUCKET=...`
  * `OBJECT_STORAGE_ENDPOINT=...`
  * `OBJECT_STORAGE_ACCESS_KEY=...`
  * `OBJECT_STORAGE_SECRET_KEY=...`

* **Binance base URL** (optional if hardcoded, but your spec lists it):

  * `BINANCE_API_BASE_URL=https://api.binance.com` (optional; can default)

If you truly plan not to use object storage in MVP, then your backend spec and upload flow need to change (right now they assume pre-signed PUT URLs).

#### 5.2.3 “Optional env vars that confuse people” (recommend marking optional or removing)

* **Apple web sign-in variables** (`APPLE_TEAM_ID`, `APPLE_KEY_ID`, `APPLE_PRIVATE_KEY_P8_BASE64`, `APPLE_SERVICE_ID`, redirects)

  * These are only required if you implement **Sign in with Apple web/code flow** (common for PWA).
  * If MVP auth is Google + Apple native + Email OTP, you may not need them immediately.
  * Recommendation:

    * either remove from `.env.example` MVP
    * or put them under a clearly labeled “Optional: Apple Web Auth (PWA)” block.

* If you aren’t shipping PWA in MVP, these should not be in the required example.

#### 5.2.4 Email sending likely needs more config

If you use Resend (or any email provider), you typically need at least:

* `RESEND_FROM_EMAIL=...` (or a configured sender identity)
* possibly `RESEND_FROM_NAME=...`

If you hardcode it, fine, but then document it.
