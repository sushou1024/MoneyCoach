### 2.3 OCR pipeline: “screenshots or trade slips” mixed together (ambiguous + inconsistent with PRD flow separation)

In your prototypes doc, Pipeline (1) says:

> Upload + OCR: Input 1–15 images (**screenshots or trade slips**)

But later you have:

* **Portfolio Updates → Delta update** via **trade slip OCR** (Flow D / S20 Delta Update)

**PRD** treats these as different flows:

* Primary scan is **portfolio holdings screenshots**
* Delta update is **trade slip** (transaction confirmation)

**Why this matters:**
Holdings screenshots and trade slips have different extraction targets, schemas, prompts, and error modes. Mixing them into one “OCR batch response” will create messy edge cases:

* A trade slip might contain **symbol/amount/price/side**, but not total value and not a holdings list.
* Holdings screenshots contain portfolio tables, but not “side” or executed price.

**Recommended fix:** Split OCR into two explicit endpoints/contracts:

* `POST /ocr/portfolio_batch` → holdings schema
* `POST /ocr/trade_slip` → slip schema

Even if you reuse the same model, keep **different system prompts + strict schemas**.

## 3) Strategy library vs MVP data reality (biggest under-spec risk)

The strategy library doc includes real parameter functions (S01, S04, S05) that require data you likely won’t have reliably from screenshots.

### 3.1 Cost basis / PnL / holding time is optional in OCR, but required by multiple MVP strategies

In PRD + your OCR schema:

* `avg_price` optional
* `pnl_percent` optional

But in Strategy Library:

* **S02 Martingale** needs PnL < -10% and remaining cash.
* **S03 Trailing stop** gating uses profit %, volatility.
* **S04 Layered take profit** requires profit ≥ 30% and holding time > 30 days.
* **S01 stop loss** uses avg_cost, pnl_pct, support levels, volatility.
* **S05 DCA** ideally uses income/savings (see below).

**Under-spec you need:** What happens when `avg_price` and/or `pnl_percent` is missing?

If you don’t define this, the system will either:

* hallucinate cost basis, or
* recommend strategies that can’t be parameterized, or
* silently omit strategies and look “weak”

**Recommended decision (explicit in spec):**

* If cost basis is missing for an asset, classify it as `cost_basis_status: unknown` and:

  * **Allow** S01 (stop loss) using only current price + volatility/support logic (but then stop-loss-from-cost can’t be computed; you need a fallback formula).
  * **Allow** S05 DCA (doesn’t require avg cost).
  * **Block or degrade** S02/S03/S04 unless user provides avg cost manually (or slip/statement that contains it).

**Add a UX hook:**
On S10 Pre-check: If cost basis missing for major holdings, show:

> “To unlock recovery/take-profit strategies, we need your average cost. Add it now (optional).”

### 3.2 S05 (DCA) in strategy doc depends on monthly income + savings rate — but MVP profile doesn’t collect it

Strategy library S05 uses:

* `user_monthly_income`
* `user_savings_rate`
* `user_has_holdings`

Your prototypes user_profile collects:

* markets, experience, style, pain_points, risk_preference
  …and nothing about income/savings rate.

So either:

* S05 cannot follow its own spec, or
* you must add at least one input signal.

**Fix options:**

* **Option A:** Add a minimal “Budget” step after quiz:

  * “How much can you invest monthly?” (range slider)
  * “Do you have stable monthly income?” (yes/no)
* **Option B (no income collection):** Redefine MVP S05 parameterization to be based on:

  * idle stablecoin/cash balance %
  * net worth
  * risk preference
  * volatility
    Example: DCA amount = min( idle_cash * k, net_worth * m ) with caps.
* **Option C:** Make S05 only a **frequency suggestion** in MVP, not an amount. (But your prototypes schema currently requires `amount`.)

Right now, your prototypes doc claims S05 will output `{amount, frequency, next_execution_at}` but doesn’t define how `amount` is computed with available inputs.

### 4.1 Missing: “as-of timestamp” + price snapshot semantics

If net worth is frozen, users may pay and then see:

* net worth that doesn’t match current prices
* strategy parameters computed with a different price snapshot (if you compute params later)

**Under-spec:** Do you freeze market data at preview time or paid time?

**Recommendation:** Add explicit fields:

* `valuation_as_of` (timestamp)
* `market_data_snapshot_id` or `prices_as_of`
* Freeze **net worth + asset allocation** at preview time to avoid “bait and switch” perception.
* In paid report, strategy params should either:

  * use the same snapshot (consistent but possibly stale), or
  * use latest prices but display both (“preview valuation as-of …, execution params as-of …”).

Right now, the doc doesn’t specify which.

## 5) OCR/Normalization under-specifications that will create edge-case bugs

### 5.1 Symbol ambiguity resolution is not fully specified in UX

You have: “If a symbol exists in multiple domains, require user confirmation in S10.”

**What’s missing:**

* How do you present the choice?

  * “BTC (Crypto)” vs “BTC (Stock)” vs “BTC (Forex)”
* What is the default?
* Do you remember the choice for future scans (user-level mapping)?

You should define:

* A disambiguation UI row with a dropdown:

  * `asset_type: crypto | stock | forex`
* And a rule: persist the user’s choice as a mapping (unless overridden).

### 5.2 Stablecoin vs cash vs fiat balance classification

PRD mentions pitfalls like OCR misreading USDT as USD; your prototypes has “Treat USD-like rows as cash only if source UI indicates base currency; otherwise map to stablecoin if crypto platform.”

**Under-spec:** You need a formal rule set:

* On Binance/OKX/MetaMask: “USD” likely means USDT/USDC, not fiat cash.
* On broker statements: “USD” is cash.
* Some platforms show “US Dollar” as a cash balance (rare but possible).

Recommend adding:

* `balance_type: fiat_cash | stablecoin | unknown`
* Rule precedence: platform_guess → UI keyword detection → user confirmation.

### 5.3 Unpriced / unrecognized assets handling

Your schema includes `unrecognized_items: []` but you don’t define:

* If a token is not in CoinMarketCap map, do you drop it? mark it unknown? use screenshot value_display?
* If a stock ticker isn’t found in Marketstack, do you drop it?

This affects net worth accuracy and trust.

Minimum spec to add:

* `valuation_status: priced | user_provided | unpriced`
* Net worth includes priced + user_provided, excludes unpriced (or includes with 0), but must show a warning.
