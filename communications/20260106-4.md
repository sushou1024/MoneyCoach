## P0 issues (must fix / clarify before implementation)

These are the ones likely to cause real build problems or MVP mismatches.

### P0‑1) “Minimum required endpoints” list is inconsistent with later requirements

In **External Data Sources (MVP)** you define a *minimum required set*, but later you **depend** on endpoints not included in that set.

**Examples:**

* You require stock symbol validation via **Marketstack `/tickers`** in Normalize step, but your minimum set lists only `/eod/latest`.
* You require FX currency list via **OER `/currencies.json`**, but minimum set lists only `/latest.json`.
* You specify charts (SC16/SC18) and correlation metrics using **historical OHLCV**, but your minimum set doesn’t include:

  * CMC `/v2/cryptocurrency/ohlcv/historical`
  * Marketstack historical `/eod` (or another historical endpoint)

**Why it matters:** devs will implement “minimum set” and then hit missing APIs mid-feature.

**Fix:** Update the minimum required set to match the features you committed to in the same doc. For example:

* Marketstack: add `/tickers` (or define an internal ticker DB fallback)
* OER: add `/currencies.json`
* If correlation + charts are in MVP: add CMC OHLCV historical + Marketstack `/eod` historical (or drop correlation/charts from MVP explicitly)

### P0‑2) Deterministic strategy parameters are stated, but not guaranteed in the paid report contract

You correctly state: “Backend computes parameters deterministically; LLM only writes rationale and copy.”

But in **Paid Report (Full)** your output schema implies that the **LLM returns numeric parameters** under `optimization_plan[].parameters` (and your example JSON shows numbers being returned).

That creates a compliance hole: the LLM could change numbers, reorder them, or “helpfully” recompute.

**Fix options (pick one and specify explicitly):**

**Option A (recommended): backend owns numbers, LLM owns copy**

* Backend computes `strategy_plan` with parameters.
* LLM output contains only:

  * `strategy_id`
  * `rationale`
  * `expected_outcome`
  * `warnings` / tone
* Backend merges the precomputed `parameters` back into the final report JSON.

**Option B: LLM echoes numbers but backend validates hard**

* Backend passes `locked_strategy_plan` into prompt.
* LLM must output *exact same parameters*.
* Backend rejects output if any numeric field differs (schema + value check), then retries or falls back.

Right now your doc reads like the LLM is responsible for emitting `optimization_plan.parameters`, which conflicts with PRD MVP gap #4.

### P0‑3) Active Portfolio lifecycle is not explicitly defined (and impacts Insights + quota)

You say:

* “Active Portfolio is the sole source for Insights”
* Free includes “1 active portfolio”
* Re-scan replaces Active and archives prior

But you never specify **when a snapshot becomes “Active.”** Key PRD nuance: it suggests the portfolio becomes “Active” after the user shows execution intent (auto-execute waitlist), though that’s not super explicit.

**This affects:**

* whether free users see Insights at all
* whether preview-only users get signal generation
* how “one active portfolio” is enforced

**Fix: add one clear rule**, e.g.:

* **Rule A:** first successful scan confirmation (SC10 Confirm) sets Active Portfolio (even for free).

  * Paid unlocks full report + “daily alpha” card + push notifications.
* **Rule B:** only paid + auto-execute intent sets Active Portfolio (harder to justify because Insights is retention driver).

Right now this is ambiguous.

## P1 issues (important; will cause UX bugs or policy/logic drift)

### P1‑1) OCR currency conversion: PRD prompt vs your “no LLM math” rule

The PRD OCR prompt explicitly asks the model to estimate USD conversions (e.g., “1 USD ≈ 7.2 CNY”), which conflicts with the PRD’s later guidance “不要让 AI 做加减法” and your prototype’s correct approach (convert in code).

**Fix:** adjust the OCR prompt/spec in the prototypes doc:

* LLM extracts **display_currency + numeric value**
* backend does FX conversion with OER
* keep a flag like `currency_converted: true` but set by backend, not LLM

(Your prototype already behaves this way; just make it explicit and aligned to avoid prompt drift.)

---

### P1‑2) Dedupe logic uses `total_value_usd_priced` which can drift

Your dedupe says:

> same platform_guess, same row count, and total_value_usd_priced matches exactly

But `value_usd_priced` is dependent on real-time quotes; two identical screenshots processed at slightly different times could produce tiny differences → dedupe fails. Or worse: different screenshots could coincidentally match when rounded.

**Fix:** dedupe using screenshot-native features:

* screenshot-extracted total (if available)
* per-row symbols+amounts hash
* a stable “portfolio fingerprint” from OCR (symbol list + amounts), not priced USD totals

You already keep `value_from_screenshot`; that’s a better anchor.

---

### P1‑3) SC10 “Edit value” behavior is underspecified (what becomes canonical?)

You allow editing:

* symbol
* amount
* value

…but you also say:

> Editing amount or screenshot value recalculates value_usd_priced with current market prices.

If user edits `value_from_screenshot`, is that just correcting OCR, or does it override valuation? Your doc implies `value_usd_priced` is source of truth and determined by market prices, but SC10 “value edit” can confuse users.

**Fix:** explicitly define:

* Canonical inputs: `symbol + amount` are canonical; `value_from_screenshot` is reference only.
* If user edits value, either:

  * (A) treat as editing `amount` indirectly (“reverse compute amount using current price”), or
  * (B) treat as just correcting the stored screenshot reference and keep priced valuation separate.

Right now it’s unclear.

### P1‑4) Strategy applicability: you may be too restrictive when avg_price missing

You state:

* If `cost_basis_status` unknown → don’t recommend S02/S03/S04 for that asset.

But many broker/exchange screenshots show **PnL%** without avg price, or show **unrealized PnL** but not explicit avg cost. You can derive avg cost from `(current_price, pnl_percent)` in many cases.

**Fix:** add derivation rules:

* If `pnl_percent` is present and `avg_price` is missing:

  * compute implied avg price deterministically in backend
  * mark `avg_price_source: derived_from_pnl_percent`
    Then S02/S03/S04 can still be used.

Also specify whether you persist user-added avg cost across scans.

### P1‑5) Risk taxonomy is not normalized between preview and paid

You require:

* preview outputs exactly 3 risks (`identified_risks`)
* paid report expands them 1:1

But you don’t define:

* allowed `type` enum list
* severity enum (you use High/Medium but not formal)
* whether risk ids are stable across re-runs

**Fix:** define a risk type enum and a severity enum:

* types: Liquidity, Concentration, Volatility, Correlation, Drawdown, etc.
* severity: Low/Medium/High/Critical (or numeric)

And define the invariant:

* paid report must include `risk_id` matching preview `id`
